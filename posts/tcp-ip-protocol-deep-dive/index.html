TCP/IP 协议解析：从数据包到网络通信

前言#

TCP/IP 是互联网的基石。当你打开浏览器访问一个网页时，背后就是一整套协议在协作完成数据传输。这篇文章会从协议栈的分层结构出发，逐层拆解 TCP/IP 的核心机制。

一、TCP/IP 四层模型#

TCP/IP 协议栈分为四层，每一层负责不同的职责：

数据从应用层发出，每经过一层都会添加该层的协议头（封装），到达目标后再逐层解封装。这种分层设计让每一层可以独立演进——比如网络层从 IPv4 升级到 IPv6，应用层的 HTTP 完全不需要修改。

二、IP 协议：网络层的核心#

IP 协议解决的核心问题是：如何把数据包从源地址送到目的地址。

2.1 IPv4 地址#

IPv4 地址是 32 位，通常用点分十进制表示，如 192.168.1.100。地址分为网络号和主机号两部分，通过子网掩码区分：

类别范围默认掩码用途
A1.0.0.0 - 126.255.255.255/8大型网络
B128.0.0.0 - 191.255.255.255/16中型网络
C192.0.0.0 - 223.255.255.255/24小型网络

实际使用中，CIDR（无类别域间路由）已经取代了传统的分类编址。10.0.0.0/8 表示前 8 位是网络号，后 24 位是主机号。

2.2 IP 数据报头部#

几个关键字段：

  • TTL (Time to Live)：每经过一个路由器减 1，减到 0 时丢弃。防止数据包在网络中无限循环。
  • Protocol：标识上层协议，6 = TCP，17 = UDP，1 = ICMP。
  • Identification + Fragment Offset：用于分片重组。当数据包超过 MTU（通常 1500 字节）时，路由器会将其分片。

2.3 路由选择#

路由器根据路由表决定数据包的下一跳。路由表的核心逻辑是最长前缀匹配：

目标地址: 192.168.1.100

路由表:
  192.168.1.0/24   → eth0  (匹配 24 位)
  192.168.0.0/16   → eth1  (匹配 16 位)
  0.0.0.0/0        → gw    (默认路由)

结果: 选择 /24 那条，因为它匹配的前缀最长。

三、TCP 协议：可靠传输#

TCP 在不可靠的 IP 层之上构建了可靠的、面向连接的、字节流传输服务。

3.1 TCP 头部#

  • Sequence Number：字节流的序号，保证数据按序到达。
  • Window：滑动窗口大小，用于流量控制。告诉对方"我还能接收多少字节"。
  • Flags：控制标志位，SYN / ACK / FIN / RST 等。

3.2 三次握手（Three-Way Handshake）#

TCP 连接建立需要三次握手：

为什么是三次？ 核心原因是防止历史重复连接初始化。假设只有两次握手：

  1. 客户端发送 SYN（由于网络延迟，迟迟未到达服务端）
  2. 客户端超时，发送新的 SYN
  3. 服务端收到新 SYN，回复 SYN+ACK，连接建立
  4. 服务端随后又收到旧 SYN，回复 SYN+ACK，又建立一个连接 —— 资源浪费

三次握手可以让客户端在第三次 ACK 时确认"这是不是我真正想建立的连接"，旧连接的 SYN 到达时客户端不会回复 ACK。

3.3 四次挥手（Four-Way Handshake）#

连接关闭需要四次挥手：

为什么需要四次？ 因为 TCP 是全双工的。FIN 只表示"我这边没有数据要发了"，但对方可能还有数据没发完。所以 ACK 和 FIN 需要分两步发送。

2MSL 等待：主动关闭方发完最后一个 ACK 后要等待 2 倍 MSL（Maximum Segment Lifetime，报文最大生存时间）。如果 ACK 丢失，被动方会重发 FIN，主动方可以重新回复 ACK。

3.4 流量控制：滑动窗口#

发送方不能无脑发数据，否则会淹没接收方。TCP 通过滑动窗口实现流量控制：

发送窗口:
┌──────────────────────────────────────────────┐
│  已发送已确认  │  已发送未确认  │   可发送    │  不可发送
└──────────────────────────────────────────────┘
                 ◄── Window Size ──►

接收方通过 ACK 报文中的 Window 字段告知自己剩余的缓冲区大小。
当 Window = 0 时，发送方暂停发送，直到收到窗口更新。

3.5 拥塞控制#

流量控制关注的是接收方的处理能力，拥塞控制关注的是整个网络的承载能力。

TCP 拥塞控制包含四个阶段：

  1. 慢启动（Slow Start）：拥塞窗口（cwnd）从 1 个 MSS 开始，每收到一个 ACK 翻倍（指数增长）
  2. 拥塞避免（Congestion Avoidance）：cwnd 达到慢启动阈值（ssthresh）后，每个 RTT 窗口加 1（线性增长）
  3. 快速重传（Fast Retransmit）：收到 3 个重复 ACK，立即重传丢失的包，不等超时
  4. 快速恢复（Fast Recovery）：将 ssthresh 减半，cwnd 设为 ssthresh，进入拥塞避免阶段
cwnd
  ^
  │        ╱  ← 慢启动（指数增长）
  │       ╱
  │      ╱ ─ ─ ─ ─ ← 拥塞避免（线性增长）
  │     ╱        ╲
  │    ╱          ─ ─ ← 快速恢复后继续拥塞避免
  │   ╱
  │──╱───────────────────────► 时间
  │   ssthresh

四、一次完整的 HTTP 请求经历了什么#

把上面的知识串起来，当你在浏览器输入 https://example.com 时：

1. DNS 解析
   浏览器缓存 → 系统缓存 → 路由器缓存 → ISP DNS → 递归查询
   得到 IP: 93.184.216.34

2. TCP 三次握手
   本机(随机端口) ←→ 93.184.216.34:443
   SYN → SYN+ACK → ACK

3. TLS 握手
   ClientHello → ServerHello → 证书验证 → 密钥交换 → Finished

4. 发送 HTTP 请求
   GET / HTTP/1.1
   Host: example.com

5. 服务器处理并返回响应
   HTTP/1.1 200 OK
   Content-Type: text/html
   <html>...</html>

6. 浏览器渲染页面

7. TCP 四次挥手（HTTP/1.1 keep-alive 可以复用连接）

整个过程中，数据从应用层出发，经过 TCP 封装（加 TCP 头）、IP 封装（加 IP 头）、以太网帧封装（加 MAC 头），通过物理链路传输，中间经过若干路由器跳转，最终到达目标服务器。

五、用 tcpdump 抓包验证#

理论知识可以用工具验证。以下是一些实用的抓包命令：

# 捕获所有 TCP 流量
sudo tcpdump -i any tcp

# 只看三次握手
sudo tcpdump -i any 'tcp[tcpflags] & (tcp-syn|tcp-ack) != 0'

# 捕获特定端口的流量
sudo tcpdump -i any port 443 -nn

# 保存到文件用 Wireshark 分析
sudo tcpdump -i any -w capture.pcap port 80

用 Wireshark 打开 .pcap 文件，可以直观地看到每一层的协议头、标志位、序列号等详细信息。

六、总结#

TCP/IP 协议栈的设计哲学可以归结为两点：

  • 分层抽象：每一层只关心自己的职责，通过标准接口与上下层交互
  • 端到端原则：网络层（IP）只提供尽力而为的交付，可靠性由端点（TCP）保证

理解这些底层机制，对于排查网络延迟、优化连接性能、设计分布式系统都有直接帮助。协议的设计中蕴含了大量工程智慧——三次握手、滑动窗口、拥塞控制，每一个机制背后都有明确要解决的问题。
