前言#

如果你在做 AI 应用——不管是 RAG、Agent 还是其他基于 LLM 的系统——你可能每天都在调用 chat.completions.create()。但你有没有想过：输入的一段文本，在模型内部到底经历了什么？为什么有些 Prompt 效果好，有些不好？为什么长上下文推理又慢又贵？

理解 LLM 的底层原理，不是为了造轮子，而是为了在调 API 的时候能做出更好的决策。这篇文章会从 Tokenization 出发，经过 Embedding、Transformer 架构、预训练，一直到推理过程，逐层拆解 LLM 的核心机制，并在每个环节讨论它对 AI 应用开发的实践意义。

一、Tokenization：文本如何进入模型#

LLM 不直接处理文本，它处理的是 Token——文本的一个子单元。Token 可以是一个词、一个子词（subword），甚至一个字符。

1.1 BPE 算法#

主流 LLM（GPT 系列、Llama 等）使用的分词算法是 BPE（Byte Pair Encoding）。核心思路很简单：把训练语料中出现频率最高的字符对不断合并，直到词汇表达到预设大小。

举个例子，假设训练语料中有：

low: 5 次
lower: 2 次
newest: 6 次
widest: 3 次

初始状态，所有词按字符拆分：l o w、l o w e r、n e w e s t、w i d e s t。

统计相邻字符对的频率，e 和 s 出现了 9 次（newest 和 widest 中各有多处），是最频繁的。合并 e + s → es。继续这个过程，最终词汇表会包含常见组合如 low、er、est 等。

BPE 的关键特性：

  • 高频词会被完整保留为一个 Token（如 the、hello）
  • 低频词会被拆成多个子词 Token（如 uncharacteristically → un + character + istic + ally）
  • 未知词不会出现 OOM（Out of Vocabulary）问题，因为最终可以拆到字符级别

1.2 不同模型的 Tokenizer 差异#

不同模型使用不同的 Tokenizer，直接影响 Token 数量和成本：

模型Tokenizer特点
GPT-4 / GPT-4ocl100k_base / o200k_base多语言支持好，中文约 1.5-2 Token/字
Llama 3Tiktoken (BPE)词表 128K，覆盖更多语言
Qwen基于 BPE中文优化，约 1 Token/字
Claude自研 Tokenizer中文约 1-1.5 Token/字

1.3 实践意义#

理解 Tokenization 对 AI 应用开发有直接帮助：

Token 计数 = 成本和延迟。API 按 Token 计费，输入 + 输出的 Token 总数直接决定费用。在 RAG 场景中，检索回来的文档塞进 Context 时，你需要精确控制 Token 数。建议用 tiktoken（OpenAI）或对应模型的 Tokenizer 库在本地预估 Token 数，而不是等到 API 返回才发现超了。

不同语言的 Token 效率差异巨大。同样的内容，英文可能只需 100 Token，中文可能要 200+。在做多语言应用时，这个差异直接影响上下文窗口的实际可用容量。

Prompt 中的空格、换行、标点都是 Token。在构建 System Prompt 时，精简不必要的格式可以省下不少 Token。

二、Embedding：从 Token 到向量#

Token 只是文本的离散标识，模型需要把 Token 映射到连续向量空间——这就是 Embedding。

2.1 Embedding 的本质#

每个 Token 会被映射为一个固定维度的向量（如 768 维、4096 维）。这个向量是模型在训练中学到的，它能编码 Token 的语义信息：

  • 语义相近的词，向量距离也近（如 “king” 和 “queen”）
  • 向量之间的运算有语义含义（如 king - man + woman ≈ queen）

Embedding 层本质上是一个查找表（Lookup Table）。Token ID → 对应的向量，就这么简单。但这些向量不是固定的——它们在训练过程中不断更新，最终编码了语言的结构和语义。

2.2 位置编码（Positional Encoding）#

Transformer 本身没有顺序概念——它同时看到所有 Token，不知道谁在谁前面。所以需要位置编码告诉模型每个 Token 的位置。

原始 Transformer 用正弦/余弦函数生成位置向量，加到 Embedding 上。后来的模型用了更灵活的方案：

  • RoPE（旋转位置编码）：Llama、Qwen 等主流模型使用，通过旋转矩阵编码相对位置关系
  • ALiBi：直接在 Attention 计算中加距离惩罚，支持外推到更长的序列

2.3 实践意义#

RAG 场景中的 Embedding 模型 ≠ LLM 的 Embedding 层。这是两个不同的东西。LLM 内部的 Embedding 是为语言建模任务优化的，而 RAG 用的 Embedding 模型（如 text-embedding-3-small、BGE、M3E）是专门为语义相似度训练的。选择合适的 Embedding 模型对检索质量有决定性影响。

上下文窗口的本质是位置编码的外推能力。为什么 GPT-4 能支持 128K 上下文，而小模型只能 4K？核心原因之一就是位置编码方案能否有效外推到更长的序列。理解这一点，你在选择模型时就能更好地权衡"上下文长度"和"实际效果"——不是所有声称支持长上下文的模型都能用好长上下文。

三、Transformer：LLM 的核心引擎#

Transformer 是目前所有主流 LLM 的基础架构。我们不需要深入每个数学细节，但核心机制必须理解。

3.1 自注意力机制（Self-Attention）#

自注意力是 Transformer 的核心创新。它的作用是：让序列中的每个 Token 都能与所有其他 Token 建立关联。

具体来说，每个 Token 会生成三个向量：

  • Q（Query）：我想找什么信息？
  • K（Key）：我有什么信息？
  • V（Value）：我的实际内容是什么？

计算过程：

Attention(Q, K, V) = softmax(Q × K^T / √d_k) × V

直观理解：Q 和 K 的点积决定"关注度"（哪些 Token 与当前 Token 最相关），softmax 归一化后作为权重，加权求和 V 得到当前 Token 的新表示。

示例: "猫坐在垫子上，它很舒服"

计算 "它" 的 Attention 时:
  "猫"    → 高权重（代词指代）
  "垫子"  → 中权重（可能的指代对象）
  "坐"    → 低权重（动词，语义关联弱）
  "很舒服" → 中权重（描述状态）

3.2 多头注意力（Multi-Head Attention）#

不是一个 Attention，而是多个 Attention 并行运行。每个"头"关注不同的模式：

  • 有的头关注语法关系（主语-谓语）
  • 有的头关注语义关联（实体-属性）
  • 有的头关注长距离依赖（句首的代词-句末的名词）

多个头的输出拼接后做一次线性变换，得到最终结果。

3.3 Transformer Block 的完整结构#

一个 Transformer Block 包含：

输入
  ↓
Multi-Head Self-Attention  ← 捕捉 Token 间的关系
  ↓
Add & Norm (残差连接 + Layer Normalization)
  ↓
Feed-Forward Network (FFN)  ← 两层全连接，增加非线性表达能力
  ↓
Add & Norm
  ↓
输出

LLM 就是把这样的 Block 堆叠很多层：

模型层数隐藏维度注意力头数
GPT-248160025
GPT-3 (175B)961228896
Llama 3 (70B)80819264

3.4 实践意义#

理解 Attention 机制有助于设计更好的 Prompt。Attention 的本质是"相关性加权"，模型在生成每个 Token 时都会回看所有前面的 Token。这就是为什么：

  • 指令放在开头或结尾效果更好——开头和结尾的 Token 位置上更容易获得高 Attention 权重（类似人类阅读的"首因效应"和"近因效应"）
  • Few-shot 示例的格式很重要——示例之间的结构一致性会让模型的 Attention 更容易捕获到正确的模式
  • 长文本中间的内容容易被"忽略"——所谓的"Lost in the Middle"现象，因为中间位置的 Token 在 Attention 计算中容易被稀释

理解 Transformer Block 有助于选择合适的模型。层数、隐藏维度、注意力头数决定了模型的能力上限和计算成本。在 Agent 场景中，如果任务以简单指令为主，70B 模型可能是浪费；如果涉及复杂推理和长文本理解，小模型可能力不从心。

四、预训练：模型如何学习语言#

LLM 的核心训练目标是 Next Token Prediction（下一个 Token 预测）：给定前面的 Token，预测下一个 Token 的概率分布。

4.1 自回归生成#

LLM 是自回归模型——每次只生成一个 Token，然后把生成的 Token 拼到输入中，再预测下一个：

输入: "今天天气"
  → 模型预测下一个 Token: "很"  (P=0.3)
  → 拼接: "今天天气很"
  → 模型预测下一个 Token: "好"  (P=0.8)
  → 拼接: "今天天气很好"
  → ...
  → 模型预测: <EOS> (结束符)

这就是为什么 LLM 的生成是流式的——每次只产出一个 Token，天然适合 SSE（Server-Sent Events）逐个推送给前端。

4.2 训练过程的三个阶段#

原始文本 → 预训练（Next Token Prediction）→ 基座模型
                                         ↓
                  SFT（监督微调）→ 指令遵循模型
                                         ↓
                  RLHF/DPO（人类偏好对齐）→ 对话模型
  • 预训练：海量无标注文本（TB 级），目标是学习语言的统计规律。这个阶段模型学会了语法、事实知识、推理模式，但不会"对话"
  • SFT（Supervised Fine-Tuning）：用人工编写的"指令-回复"对微调，让模型学会按指令行动
  • RLHF/DPO：用人类偏好数据进一步对齐，让模型更"有用、无害、诚实"

4.3 实践意义#

理解自回归生成有助于优化流式输出体验。知道模型是逐 Token 生成的，你就能理解：

  • 为什么流式输出中偶尔会出现"纠正"（模型生成了 Token A，接着又生成"不对，应该是 B"）——因为每个 Token 的预测是独立的，前面的 Token 一旦生成就无法撤回
  • 为什么 Temperature 和 Top-P 参数有效——它们控制的是从概率分布中采样的策略

理解训练阶段有助于选择和使用模型：

  • 开源的"基座模型"（如 Llama-3-70B-base）没经过 SFT，不能直接对话，需要自己做微调
  • 如果你的 Agent 需要工具调用能力，确保选择的模型经过了 Function Calling 的 SFT 训练
  • 不同模型在 RLHF 阶段的偏好不同，这直接决定了它们在特定任务上的表现差异

五、推理过程：从输入到输出#

推理（Inference）就是把训练好的模型拿来用的过程。理解推理机制对优化 AI 应用性能至关重要。

5.1 KV Cache：推理加速的关键#

自回归生成有一个明显的效率问题：生成第 N 个 Token 时，需要计算它对所有前面 N-1 个 Token 的 Attention。每一步都要重复计算前面所有 Token 的 K 和 V 向量。

KV Cache 的思路很简单：把已经算过的 K 和 V 缓存起来，下次只算新 Token 的 Q，跟缓存的 K、V 做 Attention 即可。

无 KV Cache:
  Token 1: 计算 K1, V1
  Token 2: 重新计算 K1,V1 + K2,V2
  Token 3: 重新计算 K1,V1 + K2,V2 + K3,V3
  ...每步都是 O(N²)

有 KV Cache:
  Token 1: 计算 K1, V1 → 缓存
  Token 2: 计算 K2, V2 → 追加缓存，用 K1+K2, V1+V2
  Token 3: 计算 K3, V3 → 追加缓存，用 K1+K2+K3, V1+V2+V3
  ...每步只算 O(N)

代价是显存占用。KV Cache 随序列长度线性增长，这也是长上下文推理成本高的重要原因。

5.2 采样策略#

模型输出的是所有可能 Token 的概率分布，如何从中选一个 Token，由采样策略决定：

参数作用建议值
Temperature控制随机性。T=0 确定性输出，T 越高越随机代码/ factual 任务: 0-0.3；创意任务: 0.7-1.0
Top-P只从累积概率前 P 的 Token 中采样0.9-0.95
Top-K只从概率最高的 K 个 Token 中采样较少单独使用
Frequency Penalty降低已出现 Token 的概率，减少重复0-0.5
Presence Penalty只要出现过就惩罚，鼓励多样性0-0.5

5.3 推理优化技术#

除了 KV Cache，还有很多推理优化手段：

  • Flash Attention：优化 Attention 的内存访问模式，不存中间矩阵，速度提升 2-4x
  • 量化（Quantization）：将模型权重从 FP16 降到 INT8/INT4，显存减半，速度翻倍，精度损失可控
  • 投机采样（Speculative Decoding）：用小模型快速生成候选 Token，大模型并行验证，有效加速
  • 连续批处理（Continuous Batching）：不等所有请求完成再处理下一批，有请求完成就立刻填入新请求，提高 GPU 利用率

5.4 实践意义#

理解 KV Cache 有助于优化 Agent 的上下文管理。在多轮对话的 Agent 系统中，历史消息越长，KV Cache 占用越大，推理越慢。这告诉你：

  • 不是上下文越长越好，要在"提供足够信息"和"保持推理效率"之间找平衡
  • 对话摘要（把历史消息压缩为摘要）不只是节省 Token，也是在控制推理延迟
  • 如果用开源模型自部署，KV Cache 的显存占用是你规划 GPU 资源的关键指标

采样参数的选择直接影响 Agent 的稳定性。Agent 系统中，工具调用的参数需要精确，JSON 格式不能出错：

  • Function Calling 场景建议 Temperature 设低（0-0.1），确保输出稳定可解析
  • 创意生成场景可以适当提高 Temperature，但注意加 JSON Schema 约束
  • 如果 Agent 经常重复相同动作，适当提高 Frequency Penalty

推理优化技术是自部署模型的基础知识。如果你在考虑自部署 LLM（用 vLLM、SGLang 等框架），理解量化、Flash Attention、Continuous Batching 这些概念能帮你做出更好的部署决策。

六、全景回顾#

把以上所有环节串起来，一段文本在 LLM 中的完整旅程：

用户输入: "什么是 TCP 协议？"
         ↓
  Tokenization: ["什么", "是", "TCP", "协议", "？"]
         ↓
  Embedding + 位置编码: 每个 Token → 向量 + 位置信息
         ↓
  Transformer × N层: 自注意力捕捉关系，FFN 增加表达力
         ↓
  输出概率分布: 所有候选 Token 的概率
         ↓
  采样策略: 从分布中选一个 Token（如 "TCP"）
         ↓
  自回归循环: 把生成的 Token 拼回输入，继续生成下一个
         ↓
  最终输出: "TCP（传输控制协议）是一种面向连接的、可靠的传输层协议..."

每个环节都不是黑盒——理解了 Tokenization，你能控制成本；理解了 Embedding，你能选好 RAG 的向量模型；理解了 Transformer，你能写出更有效的 Prompt；理解了推理机制，你能优化 Agent 的响应速度和资源消耗。

这些底层知识不会直接帮你写出一行代码，但会让你在面对"为什么模型这么慢"、“为什么这个 Prompt 不 work”、“该选哪个模型"这些问题时，有能力做出有依据的判断。
