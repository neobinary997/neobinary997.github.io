一、检索环节为什么值得单独讲#

上一篇我们把文档从解析、分块、向量化到入库的 Pipeline 跑通了。但入库只是准备——用户提问时，系统能否又准又全地找到相关内容，才是决定 RAG 最终效果的关键。

一个典型的检索流程：

用户提问
  ↓ 查询理解与改写（可选）
  ↓ 检索：从知识库中召回候选文档
  ↓ 重排序：精排 top-k 结果
  ↓ 上下文组装：拼装 Prompt 发给 LLM
生成回答

这篇文章聚焦检索和重排序环节，涵盖向量检索、全文检索、混合检索、Reranking、查询改写，以及检索质量评估。

二、向量检索：语义匹配的基础#

向量检索的原理：把 query 和文档都编码为向量，用余弦相似度（或内积）衡量语义距离，返回最相似的 top-k 个 chunk。

2.1 Milvus 向量检索#

from pymilvus import MilvusClient

client = MilvusClient(uri="http://localhost:19530")

def vector_search(query: str, top_k: int = 10) -> list[dict]:
    """基础向量检索"""
    query_embedding = model.encode(
        ["Represent this sentence for searching relevant passages: " + query],
        normalize_embeddings=True,
    )[0].tolist()

    results = client.search(
        collection_name="knowledge_base",
        data=[query_embedding],
        limit=top_k,
        output_fields=["content", "source_file", "heading_path", "page_number"],
    )

    return [
        {
            "content": hit["entity"]["content"],
            "score": hit["distance"],
            "source": hit["entity"]["source_file"],
            "heading": hit["entity"]["heading_path"],
            "page": hit["entity"]["page_number"],
        }
        for hit in results[0]
    ]

2.2 带元数据过滤的检索#

元数据过滤是提升检索精度的低成本手段。如果用户只关心某个部门的文档，在检索阶段直接过滤，比在 LLM 生成阶段再排除要高效得多。

results = client.search(
    collection_name="knowledge_base",
    data=[query_embedding],
    limit=10,
    filter='department == "技术部" and doc_type == "pdf"',
    output_fields=["content", "source_file", "heading_path"],
)

2.3 向量检索的局限#

向量检索擅长处理语义相似的匹配，但对以下场景效果不好：

场景问题示例
精确术语/专有名词语义匹配可能"过于泛化"查 “K8s” 找不到 “Kubernetes” 的文档
短查询缺乏上下文，向量编码不准确查 “部署方案” 召回各种无关的部署文档
否定/排除语义向量难以表达"不要"“非 Java 方案” 可能召回 Java 方案
罕见关键词训练数据少，编码质量低内部系统代号、产品特有术语

这就是为什么需要全文检索作为互补。

三、全文检索：关键词精确匹配#

全文检索基于 BM25 算法，通过词频（TF）和逆文档频率（IDF）计算文档与查询的相关性。它不依赖语义理解，但能精确匹配关键词。

3.1 Milvus 全文检索#

Milvus 2.5+ 原生支持 BM25 全文检索。入库时将文本转为稀疏向量（Sparse Vector），查询时输入原始文本即可：

from pymilvus import MilvusClient, AnnSearchRequest

client = MilvusClient(uri="http://localhost:19530")

def bm25_search(query: str, top_k: int = 10) -> list[dict]:
    """BM25 全文检索"""
    results = client.search(
        collection_name="knowledge_base",
        data=[query],
        anns_field="sparse_vector",
        param={"metric_type": "BM25"},
        limit=top_k,
        output_fields=["content", "source_file", "heading_path"],
    )

    return [
        {
            "content": hit["entity"]["content"],
            "score": hit["distance"],
            "source": hit["entity"]["source_file"],
        }
        for hit in results[0]
    ]

3.2 向量 vs 全文：互补而非替代#

维度向量检索BM25 全文检索
匹配方式语义相似度关键词匹配
同义词/近义词支持不支持
精确术语可能泛化精确匹配
短查询效果一般较好
罕见词编码质量低IDF 加权，效果好
计算方式浮点向量距离稀疏向量点积
语言理解有无

结论：生产环境应该同时使用两种检索，然后融合结果。

四、混合检索：向量 + 全文的最佳实践#

混合检索（Hybrid Search）同时执行向量检索和全文检索，然后融合两路结果。

4.1 为什么混合检索效果更好#

用户: "Milvus hybrid search 怎么配置？"

向量检索召回:
  ✓ "Milvus 支持稠密向量和稀疏向量的联合查询..."  (语义相关)
  ✗ "Elasticsearch 也支持混合检索方案..."          (语义相似但不相关)

BM25 召回:
  ✓ "Milvus hybrid search 配置指南..."             (精确关键词匹配)
  ✓ "Milvus 2.5 新增 BM25 全文检索..."             (精确关键词匹配)
  ✗ 遗漏了语义相关但没有"Milvus"关键词的文档

混合融合后:
  ✓ 两路的优点都被保留，无关结果被稀释

4.2 Milvus 混合检索实现#

Milvus 的 hybrid_search 方法原生支持多路检索 + 融合排序：

from pymilvus import AnnSearchRequest, RRFRanker

def hybrid_search(query: str, top_k: int = 10) -> list[dict]:
    """混合检索：向量 + BM25，RRF 融合"""

    # 1. 构建向量检索请求
    query_embedding = model.encode(
        ["Represent this sentence for searching relevant passages: " + query],
        normalize_embeddings=True,
    )[0].tolist()

    dense_request = AnnSearchRequest(
        data=[query_embedding],
        anns_field="vector",
        param={"metric_type": "COSINE"},
        limit=top_k * 3,  # 多召回一些，留给融合阶段筛选
    )

    # 2. 构建 BM25 全文检索请求
    sparse_request = AnnSearchRequest(
        data=[query],
        anns_field="sparse_vector",
        param={"metric_type": "BM25"},
        limit=top_k * 3,
    )

    # 3. 执行混合搜索，RRF 融合
    results = client.hybrid_search(
        collection_name="knowledge_base",
        reqs=[dense_request, sparse_request],
        ranker=RRFRanker(k=60),  # RRF 融合，k=60 是常用值
        limit=top_k,
        output_fields=["content", "source_file", "heading_path", "page_number"],
    )

    return [
        {
            "content": hit["entity"]["content"],
            "score": hit["distance"],
            "source": hit["entity"]["source_file"],
            "heading": hit["entity"]["heading_path"],
        }
        for hit in results[0]
    ]

也可以用加权融合替代 RRF：

# WeightedRanker 参数顺序对应 reqs 列表
# 0.7 给 BM25，1.0 给向量 — 偏重语义匹配

4.3 RRF 融合算法详解#

Reciprocal Rank Fusion（RRF）是最常用的多路融合策略。它的核心思路：不依赖原始分数，只用排名位置来计算融合分数。

RRF_score(doc) = Σ  1 / (k + rank_i(doc))    对每个检索器 i 求和

k 通常取 60。k 越大，排名差异的影响越小，体现的是**“共识优于单项突出”**——在多个检索器中都出现的文档比只在单路排第一的文档更可靠。

手动实现 RRF 的代码，适用于任何检索后端：

def reciprocal_rank_fusion(
    result_lists: list[list[tuple[str, float]]],
    k: int = 60,
) -> list[tuple[str, float]]:
    """
    多路检索结果融合
    result_lists: 多个检索器的结果，每个是 [(doc_id, score), ...]
    返回: [(doc_id, rrf_score), ...] 按融合分数降序
    """
    rrf_scores: dict[str, float] = {}
    for results in result_lists:
        for rank, (doc_id, _) in enumerate(results, start=1):
            rrf_scores[doc_id] = rrf_scores.get(doc_id, 0) + 1.0 / (k + rank)

    return sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)

# 使用示例
bm25_results = [("doc_A", 12.4), ("doc_B", 8.1), ("doc_C", 5.3)]
vector_results = [("doc_B", 0.92), ("doc_A", 0.85), ("doc_D", 0.78)]

fused = reciprocal_rank_fusion([bm25_results, vector_results])
# doc_A: 1/61 + 1/62 ≈ 0.0326  (两路都出现，融合分最高)
# doc_B: 1/62 + 1/61 ≈ 0.0324
# doc_C: 1/63 ≈ 0.0159
# doc_D: 1/63 ≈ 0.0159

RRF vs 加权融合：

维度RRFWeighted Ranker
调参几乎不需要（k=60 通用）需要调权重
鲁棒性高，不受分数分布影响权重不合理时效果差
精细控制有限可精确调整各路贡献
适用场景通用默认方案对某路检索有明显偏好时

实践建议： 先用 RRF 作为默认方案，在评估数据集上对比 Weighted Ranker 的效果，有明显提升再切换。

五、Reranking：精排提升最终质量#

检索阶段的 bi-encoder（如 BGE）追求速度，把 query 和文档独立编码再算相似度。Reranking 阶段用 cross-encoder，把 query 和每个候选文档拼接后联合编码，精度更高但速度更慢。

5.1 Reranking 在 Pipeline 中的位置#

混合检索召回 top-30 候选
  ↓
Reranker 精排 → 输出 top-5
  ↓
拼入 Prompt → LLM 生成回答

先用检索阶段多召回一些（top-20 ~ top-30），再由 Reranker 精排到 top-5 送入 LLM。这样兼顾了召回率和精度。

5.2 BGE-Reranker（开源推荐）#

BAAI/bge-reranker-v2-m3 支持中文，精度和效率均衡：

from FlagEmbedding import FlagReranker

reranker = FlagReranker("BAAI/bge-reranker-v2-m3", use_fp16=True)

def rerank(query: str, candidates: list[dict], top_k: int = 5) -> list[dict]:
    """对检索结果做 Reranking"""
    pairs = [[query, c["content"]] for c in candidates]
    scores = reranker.compute_score(pairs, normalize=True)

    # 按分数降序排列
    scored_candidates = list(zip(candidates, scores))
    scored_candidates.sort(key=lambda x: x[1], reverse=True)

    results = []
    for candidate, score in scored_candidates[:top_k]:
        result = {**candidate, "rerank_score": score}
        results.append(result)

    return results

5.3 Cohere Rerank（托管 API）#

不想自部署模型的可以用 Cohere 的 Rerank API，支持 100+ 语言，32k 上下文窗口：

import cohere

co = cohere.ClientV2("YOUR_API_KEY")

def rerank_with_cohere(query: str, documents: list[str], top_n: int = 5):
    results = co.v2.rerank(
        model="rerank-v4.0-pro",
        query=query,
        documents=documents,
        top_n=top_n,
    )
    return [(r.index, r.relevance_score) for r in results.results]

5.5 LLM 做 Reranker#

用 LLM 对候选文档列表排序，精度最高但成本也最高。适合候选量小（5-10 个）且对排序质量要求极高的场景：

import json
from openai import OpenAI

client = OpenAI()

def llm_rerank(query: str, candidates: list[dict], top_k: int = 5) -> list[dict]:
    """用 LLM 对候选文档排序"""
    docs_text = "\n".join(
        f"[{i}] {c['content'][:200]}" for i, c in enumerate(candidates)
    )

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0,
        messages=[
            {"role": "system", "content": "你是文档相关性排序专家。只输出 JSON。"},
            {"role": "user", "content": f"""请根据查询的相关性对以下文档排序，返回最相关的 {top_k} 个文档编号。

查询: {query}

候选文档:
{docs_text}

返回格式: {{"ranking": [文档编号, ...]}}"""},
        ],
    )

    ranking = json.loads(response.choices[0].message.content)["ranking"]
    return [candidates[i] for i in ranking[:top_k]]

六、查询改写：让检索更好理解用户意图#

用户的原始查询往往太短、太模糊，或者措辞与文档不匹配。查询改写技术通过改写或扩展查询来弥合这个语义鸿沟。

6.1 HyDE：假想文档嵌入#

HyDE（Hypothetical Document Embeddings）的思路很巧妙：让 LLM 先生成一个"假想答案"，然后对这个假想答案做 Embedding 去检索真实文档。 因为"用文档检索文档"比"用问题检索文档"效果更好。

用户: "Milvus hybrid search 怎么配置？"
  ↓ LLM 生成假想答案:
  "Milvus 的混合检索通过 hybrid_search API 实现，需要同时定义
   dense 和 sparse 两路检索请求，使用 RRFRanker 融合结果..."
  ↓ 对假想答案做 Embedding
  ↓ 用这个 Embedding 去向量检索
  ↓ 召回与假想答案相似的真实文档

from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate

hyde_prompt = ChatPromptTemplate.from_template(
    "请写一段文字来回答以下问题，即使你不确定也要给出你认为最合理的回答：\n\n{question}"
)

def hyde_search(query: str, top_k: int = 5) -> list[dict]:
    """HyDE 查询：先让 LLM 生成假想答案，再检索"""
    # 1. 生成假想文档
    chain = hyde_prompt | llm | StrOutputParser()
    hypothetical_doc = chain.invoke({"question": query})

    # 2. 用假想文档的 Embedding 检索
    hyde_embedding = model.encode(
        ["Represent this sentence for searching relevant passages: " + hypothetical_doc],
        normalize_embeddings=True,
    )[0].tolist()

    results = client.search(
        collection_name="knowledge_base",
        data=[hyde_embedding],
        limit=top_k,
        output_fields=["content", "source_file", "heading_path"],
    )

    return [
        {"content": hit["entity"]["content"], "score": hit["distance"]}
        for hit in results[0]
    ]

适用场景： 查询短且模糊、技术文档检索、文档措辞与用户提问差异大的场景。

注意事项： 如果 LLM 对领域的理解偏差较大，假想答案会误导检索方向。建议与 Reranking 配合使用——HyDE 扩大召回，Reranker 兜底精度。

6.2 Multi-Query：多查询扩展#

Multi-Query 让 LLM 将原始查询改写为多个不同角度的子查询，分别检索后合并去重：

from langchain.retrievers.multi_query import MultiQueryRetriever

retriever = MultiQueryRetriever.from_llm(
    retriever=vectorstore.as_retriever(search_kwargs={"k": 5}),
    llm=llm,
)

# 自动生成 3-5 个查询变体并合并结果
docs = retriever.invoke("如何部署 Milvus 集群？")
# 可能的子查询:
#   1. "Milvus 集群部署步骤和配置"
#   2. "Milvus 分布式架构搭建指南"
#   3. "Milvus 集群环境要求和依赖"

# 手动实现，更可控
def multi_query_search(query: str, top_k: int = 5) -> list[dict]:
    """多查询检索"""
    # 1. 生成查询变体
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        temperature=0.3,
        messages=[
            {"role": "system", "content": "你是查询改写专家。"},
            {"role": "user", "content": f"""将以下查询改写为 3 个不同角度的子查询，每行一个，只输出查询：
{query}"""},
        ],
    )

    sub_queries = response.choices[0].message.content.strip().split("\n")
    sub_queries = [q.strip() for q in sub_queries if q.strip()]
    sub_queries.append(query)  # 保留原始查询

    # 2. 分别检索
    all_results: dict[str, dict] = {}
    for q in sub_queries:
        hits = vector_search(q, top_k=top_k)
        for hit in hits:
            doc_id = hit["content"][:100]  # 用内容前缀做去重 key
            if doc_id not in all_results or hit["score"] > all_results[doc_id]["score"]:
                all_results[doc_id] = hit

    # 3. 按分数排序
    results = sorted(all_results.values(), key=lambda x: x["score"], reverse=True)
    return results[:top_k]

6.3 查询改写策略对比#

策略原理LLM 调用检索次数适用场景
HyDE生成假想答案替代查询1 次1 次查询短、领域知识充分
Multi-Query生成多个查询变体1 次N 次查询模糊、需要广覆盖
Step-back生成更抽象的高层问题1 次2 次需要背景知识支撑
不改写直接用原始查询检索0 次1 次查询已经足够清晰

实践建议： 不要默认开启查询改写——先用原始查询评估检索质量，效果不够再加。优先级：HyDE（精度提升）> Multi-Query（召回提升）> Step-back（特定场景）。

七、完整的检索 Pipeline#

把上述环节组装成端到端的检索 Pipeline：

class RetrievalPipeline:
    def __init__(self, top_k: int = 5, use_reranker: bool = True, use_hyde: bool = False):
        self.top_k = top_k
        self.use_reranker = use_reranker
        self.use_hyde = use_hyde
        self.reranker = FlagReranker("BAAI/bge-reranker-v2-m3", use_fp16=True) if use_reranker else None

    def retrieve(self, query: str, filters: dict = None) -> list[dict]:
        """完整的检索流程"""
        # 1. 查询改写（可选）
        search_query = self._rewrite_query(query) if self.use_hyde else query

        # 2. 混合检索
        candidates = hybrid_search(search_query, top_k=self.top_k * 4, filters=filters)

        # 3. Reranking（可选）
        if self.use_reranker and candidates:
            results = rerank(query, candidates, top_k=self.top_k)
        else:
            results = candidates[:self.top_k]

        return results

    def _rewrite_query(self, query: str) -> str:
        """HyDE 查询改写"""
        chain = hyde_prompt | llm | StrOutputParser()
        return chain.invoke({"question": query})

八、检索质量评估#

无法衡量就无法优化。检索质量需要系统化的评估机制。

8.1 核心指标#

指标关注点计算方式
Hit Rate前K结果中是否有相关文档命中的查询数 / 总查询数
MRR第一个相关文档排多前avg(1 / 第一个相关文档的排名)
Recall@K前K结果召回了多少相关文档前K中相关文档数 / 总相关文档数
NDCG@K排序质量（高相关排前面得分更高）DCG@K / 理想DCG@K

指标选择：

目标用什么指标
快速判断检索是否可用Hit Rate
调优检索器参数MRR + Recall@K
调优 RerankerNDCG@K
端到端 RAG 质量评估用 Ragas（见下文）

8.2 评估数据集构建#

评估需要"问题 - 正确文档"的标注数据。可以从已有文档中自动生成：

import json
from openai import OpenAI

client = OpenAI()

def generate_eval_dataset(chunks: list[dict], num_questions: int = 3) -> list[dict]:
    """从 chunk 自动生成评估用问答对"""
    eval_data = []

    for chunk in chunks:
        response = client.chat.completions.create(
            model="gpt-4o-mini",
            temperature=0.3,
            messages=[{
                "role": "user",
                "content": f"""基于以下文档内容，生成 {num_questions} 个用户可能会问的问题。
问题应该能通过这段内容直接回答。严格按 JSON 格式返回。

文档内容:
{chunk["content"]}

返回格式: {{"questions": ["问题1", "问题2", ...]}}"""
            }],
        )

        questions = json.loads(response.choices[0].message.content)["questions"]
        for q in questions:
            eval_data.append({
                "question": q,
                "relevant_chunk_id": chunk.get("chunk_id", ""),
                "relevant_content": chunk["content"],
            })

    return eval_data

8.3 用 Ragas 评估端到端质量#

Ragas 是 RAG 评估的主流框架，关注四个核心维度：

Ragas 指标评估什么
Faithfulness（忠实度）回答是否基于检索到的上下文（不编造）
Answer Relevancy（答案相关性）回答是否真正回答了问题
Context Precision（上下文精确度）相关 chunk 是否排在非相关 chunk 前面
Context Recall（上下文召回）检索的上下文是否覆盖了回答所需信息
# pip install ragas
from ragas import evaluate
from ragas.metrics import Faithfulness, AnswerRelevancy, ContextPrecision, ContextRecall
from ragas.dataset_schema import SingleTurnSample, EvaluationDataset

samples = [
    SingleTurnSample(
        user_input="Milvus 的混合检索如何配置？",
        response="Milvus 的混合检索通过 hybrid_search 方法实现...",
        retrieved_contexts=["Milvus hybrid_search API 需要 AnnSearchRequest..."],
        reference="使用 hybrid_search + AnnSearchRequest + RRFRanker。",
    )
]

eval_dataset = EvaluationDataset(samples=samples)

results = evaluate(
    eval_dataset,
    metrics=[Faithfulness(), AnswerRelevancy(), ContextPrecision(), ContextRecall()],
)
print(results)

8.4 评估驱动的优化循环#

构建评估数据集
  ↓ 基线评估（当前 Pipeline 的指标）
  ↓ 调整一个变量（分块策略 / 检索参数 / Reranker）
  ↓ 重新评估，对比指标变化
  ↓ 确认有效后合入
持续迭代

每次只调整一个变量，用评估数据集量化效果。常见的优化路径：

  1. 基线：纯向量检索 + RecursiveCharacterTextSplitter
  2. +混合检索：加入 BM25，用 RRF 融合 → Hit Rate 通常提升 10-20%
  3. +Reranker：加入 BGE-reranker → NDCG 通常提升 15-30%
  4. +查询改写：对效果差的 case 加 HyDE → 长尾查询改善
  5. +分块优化：换结构感知分块 / LLM 分块 → 整体质量提升

九、总结#

检索环节的核心决策点：

环节核心决策影响范围
检索方式纯向量 vs 混合检索召回覆盖面
融合策略RRF vs 加权多路结果的排序质量
Reranking用不用、用什么模型最终 top-k 的精度
查询改写HyDE / Multi-Query / 不改写短查询和模糊查询的效果
评估指标选择、数据集构建优化方向的判断依据

几个实践中的经验判断：

  • 混合检索是性价比最高的单一改进。 从纯向量切换到向量+BM25 的 RRF 融合，通常比换一个更贵的 Embedding 模型效果更好
  • Reranker 是精排的标配。 开源的 BGE-reranker-v2-m3 就能在大多数场景下带来显著提升
  • 查询改写不是必须的。 先用原始查询评估效果，确有需要再开启
  • 评估先行。 没有评估数据集就不要调参——你无法判断一个改动是提升还是退化
