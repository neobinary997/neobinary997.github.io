一、文档处理是 RAG 的地基#

RAG（Retrieval-Augmented Generation）的核心思路很简单：把外部知识喂给 LLM，让它基于事实回答问题，而不是凭记忆编造。 但"喂知识"这一步的工程复杂度常被低估。

一个典型的企业知识库场景：技术文档是 Confluence 上的 HTML，产品方案用 Word 写，财务报表是 Excel，培训材料是 PPT，还有些老系统留下的 PDF 扫描件。格式各异、结构不同、质量参差。

文档处理 Pipeline 的目标就是把这些杂乱文档变成可检索的结构化知识：

原始文档（PDF/Word/Excel/...）
  ↓ 解析：提取文本、表格、图片
  ↓ 清洗：去噪、规范化
  ↓ 分块：切成语义完整的片段
  ↓ 向量化：Embedding 模型编码
  ↓ 入库：写入向量数据库
结构化知识库 → 支撑高质量检索

每个环节都有值得展开的决策点。这篇文章从实战角度逐个拆解。

二、文档解析：从各种格式中提取结构化内容#

文档解析的目标是把不同格式的文件统一成结构化文本 + 元数据。难点不在于"能提取多少文本"，而在于保留文档的语义结构——标题层级、段落关系、表格语义，这些信息对后续分块至关重要。

2.1 统一输出格式#

先定义一个统一的解析结果结构，所有格式的解析器都输出这个格式：

from dataclasses import dataclass, field
from typing import Optional

@dataclass
class ParsedDocument:
    """所有格式解析器的统一输出"""
    source: str                          # 文件路径或 URL
    title: str                           # 文档标题
    sections: list["DocumentSection"]    # 结构化内容
    metadata: dict = field(default_factory=dict)

@dataclass
class DocumentSection:
    """文档中的一个逻辑段落"""
    content: str                         # 文本内容
    level: int = 0                       # 标题层级，0 = 普通段落
    section_type: str = "text"           # text | table | list | image
    page_number: Optional[int] = None    # 来源页码
    metadata: dict = field(default_factory=dict)

有了这个统一抽象，下游的分块和入库逻辑不需要关心文档来源。

2.2 PDF 解析#

PDF 是企业场景最常见的格式，也是解析难度最高的。

文本型 PDF（可选中文字的）用 PyMuPDF 提取文本和表格：

import fitz  # PyMuPDF

def parse_pdf(file_path: str) -> ParsedDocument:
    doc = fitz.open(file_path)
    sections = []

    for page_num, page in enumerate(doc):
        # 提取文本，保留字体大小信息用于识别标题
        blocks = page.get_text("dict")["blocks"]
        for block in blocks:
            if block["type"] == 0:  # 文本块
                for line in block["lines"]:
                    text = "".join(span["text"] for span in line["spans"]).strip()
                    if not text:
                        continue
                    # 根据字号判断是否为标题（启发式）
                    font_size = line["spans"][0]["size"]
                    level = _infer_heading_level(font_size, page.rect.width)
                    sections.append(DocumentSection(
                        content=text,
                        level=level,
                        page_number=page_num + 1,
                    ))

        # 提取表格
        tables = page.find_tables()
        for table in tables:
            table_text = _table_to_markdown(table.extract())
            if table_text:
                sections.append(DocumentSection(
                    content=table_text,
                    section_type="table",
                    page_number=page_num + 1,
                ))

    return ParsedDocument(source=file_path, title=_extract_title(sections), sections=sections)

扫描型 PDF 需要先 OCR 再提取。PaddleOCR 在中文场景下效果不错：

from paddleocr import PaddleOCR

ocr = PaddleOCR(use_angle_cls=True, lang="ch")

def parse_scanned_pdf(file_path: str) -> ParsedDocument:
    doc = fitz.open(file_path)
    sections = []
    for page_num, page in enumerate(doc):
        # 渲染为图片
        pix = page.get_pixmap(dpi=300)
        img_bytes = pix.tobytes("png")
        # OCR 识别
        result = ocr.ocr(img_bytes, cls=True)
        page_text = "\n".join(line[1][0] for line in result[0] if line[1][1] > 0.7)
        if page_text:
            sections.append(DocumentSection(content=page_text, page_number=page_num + 1))
    return ParsedDocument(source=file_path, title=_extract_title(sections), sections=sections)

PDF 解析的实践经验：

问题策略
多栏排版导致文本顺序错乱用坐标排序，按阅读顺序（从上到下、从左到右）重排
页眉页脚反复出现按 Top N 频率匹配，识别后过滤
表格跨页合并连续的 table 类型 Section，检查表头是否重复
混合排版（文字 + 扫描图）先提取文本，对图片区域单独 OCR，按坐标位置合并

2.3 Word/Docx 解析#

Word 文档的结构信息比 PDF 好提取得多，python-docx 直接提供了段落样式：

from docx import Document

def parse_docx(file_path: str) -> ParsedDocument:
    doc = Document(file_path)
    sections = []

    for para in doc.paragraphs:
        text = para.text.strip()
        if not text:
            continue
        # 通过样式名推断标题层级
        style_name = para.style.name.lower()
        level = 0
        if "heading" in style_name:
            try:
                level = int(style_name.replace("heading", "").strip())
            except ValueError:
                level = 1
        sections.append(DocumentSection(content=text, level=level))

    # 提取表格
    for table in doc.tables:
        table_text = _table_to_markdown([[cell.text for cell in row.cells] for row in table.rows])
        if table_text:
            sections.append(DocumentSection(content=table_text, section_type="table"))

    return ParsedDocument(source=file_path, title=_extract_title(sections), sections=sections)

Word 解析的注意事项：用户经常用"加粗 + 放大字体"代替标题样式。遇到 level=0 但字号明显大于正文的段落，可以按字体大小启发式推断层级。

2.4 Excel 解析#

Excel 的挑战不在提取（openpyxl 足够），而在于理解表格的语义——一行数据没有上下文意义，需要带上列名甚至 sheet 名称。

import openpyxl

def parse_excel(file_path: str) -> ParsedDocument:
    wb = openpyxl.load_workbook(file_path, read_only=True, data_only=True)
    sections = []

    for sheet_name in wb.sheetnames:
        ws = wb[sheet_name]
        # Sheet 名称作为一级标题
        sections.append(DocumentSection(content=sheet_name, level=1))

        rows = list(ws.iter_rows(values_only=True))
        if not rows:
            continue

        # 将每行转为"列名: 值"的文本，保留语义
        headers = [str(h) if h else f"col_{i}" for i, h in enumerate(rows[0])]
        for row in rows[1:]:
            if all(cell is None for cell in row):
                continue
            line_parts = [f"{headers[i]}: {row[i]}" for i in range(len(headers)) if row[i] is not None]
            sections.append(DocumentSection(
                content="; ".join(line_parts),
                section_type="table",
                metadata={"sheet": sheet_name},
            ))

    return ParsedDocument(source=file_path, title=_extract_title(sections), sections=sections)

对于行数很多的 Excel（如日志表、明细表），逐行入库会导致检索结果碎片化。实践中按业务逻辑聚合：比如按订单号分组，将同一订单的多行合并成一段文本。

2.5 PPT 解析#

PPT 的文本量通常不大，但演讲者备注（Notes）往往包含比幻灯片正文更详细的解释，不要遗漏。

from pptx import Presentation

def parse_pptx(file_path: str) -> ParsedDocument:
    prs = Presentation(file_path)
    sections = []

    for i, slide in enumerate(prs.slides):
        page_title = f"幻灯片 {i + 1}"

        # 提取正文
        for shape in slide.shapes:
            if shape.has_text_frame:
                for para in shape.text_frame.paragraphs:
                    text = para.text.strip()
                    if text:
                        sections.append(DocumentSection(content=text, page_number=i + 1))

        # 提取备注（通常比正文更有信息量）
        if slide.has_notes_slide:
            notes = slide.notes_slide.notes_text_frame.text.strip()
            if notes:
                sections.append(DocumentSection(
                    content=f"[备注] {notes}",
                    section_type="text",
                    page_number=i + 1,
                ))

    return ParsedDocument(source=file_path, title=_extract_title(sections), sections=sections)

2.6 HTML / Markdown 解析#

内部 Wiki（Confluence、Notion）通常提供 HTML 或 Markdown 导出。HTML 用 BeautifulSoup 提取，Markdown 直接按标题分割。

from bs4 import BeautifulSoup

def parse_html(html_content: str) -> ParsedDocument:
    soup = BeautifulSoup(html_content, "html.parser")

    # 移除导航栏、侧边栏、脚本等噪声
    for tag in soup(["nav", "footer", "script", "style", "header"]):
        tag.decompose()

    sections = []
    for tag in soup.find_all(["h1", "h2", "h3", "h4", "p", "li", "pre", "table"]):
        if tag.name.startswith("h"):
            level = int(tag.name[1])
            sections.append(DocumentSection(content=tag.get_text(strip=True), level=level))
        elif tag.name == "table":
            rows = [[td.get_text(strip=True) for td in tr.find_all(["td", "th"])] for tr in tag.find_all("tr")]
            sections.append(DocumentSection(content=_table_to_markdown(rows), section_type="table"))
        else:
            text = tag.get_text(strip=True)
            if text:
                sections.append(DocumentSection(content=text))

    return ParsedDocument(source="html", title=_extract_title(sections), sections=sections)

2.8 辅助工具函数#

上面用到的几个工具函数：

import re

def _table_to_markdown(rows: list[list[str]]) -> str:
    """将二维列表转为 Markdown 表格"""
    if not rows:
        return ""
    lines = []
    for i, row in enumerate(rows):
        line = "| " + " | ".join(str(cell).replace("\n", " ") for cell in row) + " |"
        lines.append(line)
        if i == 0:
            lines.append("| " + " | ".join("---" for _ in row) + " |")
    return "\n".join(lines)

def _extract_title(sections: list[DocumentSection]) -> str:
    """从解析结果中提取文档标题（取第一个标题层级的内容）"""
    for s in sections:
        if s.level >= 1:
            return s.content
    return sections[0].content[:50] if sections else "Untitled"

def _infer_heading_level(font_size: float, page_width: float) -> int:
    """根据字号推断标题层级（启发式，需根据实际文档调整阈值）"""
    # 通常正文字号在 10-12pt，标题 14pt+
    if font_size >= 20:
        return 1
    elif font_size >= 16:
        return 2
    elif font_size >= 14:
        return 3
    return 0

三、文本清洗与规范化#

解析出的原始文本通常带着各种噪声，清洗的目标是在去除噪声的同时保留语义结构。

3.1 常见噪声类型#

噪声类型示例处理方式
页眉页脚“公司内部文件 - 第 3 页”频率统计 + 模式匹配删除
水印文字“CONFIDENTIAL”、“机密”关键词过滤
重复空行/空格连续多个 \n\n\n正则压缩
编码乱码ï»¿、﻿UTF-8 BOM 清理
目录文本“1.1 概述 … 3”（页码）检测目录模式，跳过或单独处理
无意义字符“·"、单独的序号长度阈值过滤

3.2 清洗 Pipeline#

import re

class TextCleaner:
    def __init__(self):
        self.header_footer_patterns: list[str] = []  # 运行时学习
        self.noise_keywords = {"机密", "CONFIDENTIAL", "内部文件", "仅供参考"}

    def clean(self, text: str) -> str:
        text = self._remove_bom(text)
        text = self._normalize_whitespace(text)
        text = self._remove_noise_keywords(text)
        text = self._remove_page_numbers(text)
        return text.strip()

    def learn_header_footer(self, pages: list[str]):
        """从多页文本中学习公共页眉页脚"""
        from collections import Counter
        line_counter = Counter()
        for page in pages:
            lines = page.strip().split("\n")
            if lines:
                line_counter[lines[0].strip()] += 1  # 首行
                line_counter[lines[-1].strip()] += 1  # 末行
        # 出现在超过 50% 页面中的行视为页眉/页脚
        threshold = len(pages) * 0.5
        self.header_footer_patterns = [line for line, count in line_counter.items()
                                       if count >= threshold and len(line) > 3]

    def _remove_bom(self, text: str) -> str:
        return text.replace("﻿", "").replace("�", "")

    def _normalize_whitespace(self, text: str) -> str:
        text = re.sub(r"[ \t]+", " ", text)           # 合并空格
        text = re.sub(r"\n{3,}", "\n\n", text)         # 最多保留一个空行
        return text

    def _remove_noise_keywords(self, text: str) -> str:
        for kw in self.noise_keywords:
            text = text.replace(kw, "")
        return text

    def _remove_page_numbers(self, text: str) -> str:
        # 匹配 "第 X 页" "- X -" 等页码模式
        text = re.sub(r"[-—]\s*\d+\s*[-—]", "", text)
        text = re.sub(r"第\s*\d+\s*页", "", text)
        return text

3.3 文档层级保留#

清洗时不要把所有文本压成纯平字符串。标题层级（H1 → H2 → H3）在后续分块时是天然的边界信号——保留 DocumentSection.level 信息，分块策略可以据此做出更合理的切割。

四、分块策略：决定检索质量的关键环节#

分块（Chunking）是把长文档切成较小片段的过程。分块的质量直接决定了检索的精度：块太大，噪声多，检索不够精准；块太小，上下文丢失，检索到的片段无法独立回答问题。

4.1 五种主流分块策略#

固定长度分块#

最简单的方案：按字符数切，带重叠窗口。

def fixed_size_chunk(text: str, chunk_size: int = 500, overlap: int = 50) -> list[str]:
    """按固定字符数分块，带重叠"""
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        chunk = text[start:end]
        if chunk.strip():
            chunks.append(chunk.strip())
        start += chunk_size - overlap
    return chunks

优点：实现简单，块大小均匀。缺点：完全无视语义边界，可能在句子中间切断。

递归字符分块#

LangChain 的 RecursiveCharacterTextSplitter 是目前最常用的方案。它按优先级依次尝试不同的分隔符（\n\n → \n → 。 → 空格 → 字符），尽可能在自然语义边界切分。

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", "；", " ", ""],
    length_function=len,
)

chunks = splitter.split_text(text)

适合大多数场景的"够用"方案。但对于结构化文档（技术文档、API 文档），它仍然可能在章节中间切断。

文档结构感知分块#

利用解析阶段保留的标题层级信息，按章节边界切分。这是结构化文档最推荐的方案：

from dataclasses import dataclass

@dataclass
class Chunk:
    content: str
    metadata: dict

def structure_aware_chunk(
    sections: list[DocumentSection],
    max_chunk_size: int = 800,
    min_chunk_size: int = 100,
) -> list[Chunk]:
    """按文档标题层级分块，保持章节语义完整性"""
    chunks: list[Chunk] = []
    current_chunk_parts: list[str] = []
    current_size = 0
    current_path: list[str] = []  # 当前标题路径，如 ["第三章", "3.2 部署方案"]

    for section in sections:
        text = section.content.strip()
        if not text:
            continue

        # 标题：更新层级路径
        if section.level > 0:
            # 截断到当前层级，追加新标题
            current_path = current_path[: section.level - 1]
            current_path.append(text)

            # 如果当前积累的内容已经足够大，先输出一个 chunk
            if current_size >= min_chunk_size:
                chunks.append(_build_chunk(current_chunk_parts, current_path))
                current_chunk_parts = []
                current_size = 0

            # 把标题本身也加入内容（检索时标题是重要的匹配信号）
            current_chunk_parts.append(text)
            current_size += len(text)
            continue

        # 普通段落：累积
        if current_size + len(text) > max_chunk_size and current_size >= min_chunk_size:
            chunks.append(_build_chunk(current_chunk_parts, current_path))
            current_chunk_parts = []
            current_size = 0

        current_chunk_parts.append(text)
        current_size += len(text)

    # 处理尾部
    if current_chunk_parts:
        chunks.append(_build_chunk(current_chunk_parts, current_path))

    return chunks


def _build_chunk(parts: list[str], heading_path: list[str]) -> Chunk:
    """构建一个 chunk，附带标题路径作为元数据"""
    content = "\n".join(parts)
    return Chunk(
        content=content,
        metadata={
            "heading_path": " > ".join(heading_path),
            "heading_depth": len(heading_path),
        },
    )

这个方案的核心思路：标题是天然的语义边界。遇到新标题时，如果前一个 chunk 已经有足够内容，就在标题前切一刀。这样每个 chunk 都是一个相对完整的语义单元，且带着标题路径元数据，后续检索时可以用作过滤或 reranking 信号。

语义分块#

基于 Embedding 相似度来决定切割点：相邻句子语义相似就合并，语义断裂就切分。

import numpy as np
from sentence_transformers import SentenceTransformer

model = SentenceTransformer("BAAI/bge-small-zh-v1.5")

def semantic_chunk(text: str, threshold: float = 0.5) -> list[str]:
    """基于语义相似度的分块"""
    # 按句子拆分
    sentences = re.split(r"(?<=[。！？；\n])", text)
    sentences = [s.strip() for s in sentences if s.strip()]

    if len(sentences) <= 1:
        return [text]

    # 计算相邻句子的语义相似度
    embeddings = model.encode(sentences, normalize_embeddings=True)
    similarities = [
        np.dot(embeddings[i], embeddings[i + 1])
        for i in range(len(embeddings) - 1)
    ]

    # 相似度低于阈值处切分
    chunks = []
    current = [sentences[0]]
    for i, sim in enumerate(similarities):
        if sim < threshold:
            chunks.append("".join(current))
            current = []
        current.append(sentences[i + 1])

    if current:
        chunks.append("".join(current))

    return chunks

语义分块效果很好，但成本也较高——每个句子都需要一次 Embedding 调用。适合文档量不大、对检索精度要求极高的场景。

LLM 辅助分块#

前面的方案都是"规则驱动”——用字符数、分隔符、相似度阈值来决定切分点。但如果文档量有限（几百篇以内），可以让 LLM 直接理解文档语义，自主决定分块边界。

LLM 分块有两种模式：

模式 A — 边界检测： LLM 阅读文档后返回切割点位置，应用层按位置切分。

模式 B — 分块 + 丰富： LLM 同时完成切分、摘要生成和主题标签提取，一次调用产出高质量的 chunk 及其元数据。

import json
from openai import OpenAI

client = OpenAI()

CHUNK_PROMPT = """你是一个文档分块专家。请将以下文档内容切分为语义完整的片段。

要求：
1. 每个片段应该是一个可以独立理解的语义单元，不依赖上下文也能明白在讲什么
2. 每个片段控制在 200-600 字之间
3. 不要在句子中间切断
4. 对每个片段生成：
   - content: 片段原文
   - summary: 一句话摘要（30字以内）
   - topics: 2-5 个主题关键词

请严格按以下 JSON 格式返回：
{
  "chunks": [
    {
      "content": "片段文本",
      "summary": "一句话摘要",
      "topics": ["关键词1", "关键词2"]
    }
  ]
}

文档内容：
{text}"""


def llm_chunk(text: str, max_input_chars: int = 12000) -> list[Chunk]:
    """使用 LLM 进行语义分块，同时生成摘要和主题标签"""
    # 文档过长时分段处理
    if len(text) > max_input_chars:
        return _llm_chunk_long_document(text, max_input_chars)

    response = client.chat.completions.create(
        model="gpt-4o",
        temperature=0.1,
        messages=[
            {"role": "system", "content": "你是一个文档分块专家，只输出 JSON 格式的结果。"},
            {"role": "user", "content": CHUNK_PROMPT.format(text=text)},
        ],
    )

    try:
        result = json.loads(response.choices[0].message.content)
        return [
            Chunk(
                content=c["content"],
                metadata={
                    "summary": c["summary"],
                    "topics": c["topics"],
                    "chunk_method": "llm",
                },
            )
            for c in result["chunks"]
        ]
    except (json.JSONDecodeError, KeyError) as e:
        # LLM 输出解析失败时降级到递归字符分块
        print(f"LLM 分块输出解析失败，降级到递归字符分块: {e}")
        return _fallback_chunk(text)


def _llm_chunk_long_document(text: str, max_chars: int) -> list[Chunk]:
    """长文档分段处理：先按自然段落粗切，每段独立 LLM 分块"""
    paragraphs = text.split("\n\n")
    segments: list[str] = []
    current = ""

    for para in paragraphs:
        if len(current) + len(para) > max_chars and current:
            segments.append(current.strip())
            current = para
        else:
            current += "\n\n" + para

    if current.strip():
        segments.append(current.strip())

    all_chunks = []
    for seg in segments:
        all_chunks.extend(llm_chunk(seg, max_chars))

    return all_chunks


def _fallback_chunk(text: str, chunk_size: int = 500) -> list[Chunk]:
    """降级方案：递归字符分块"""
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=50,
        separators=["\n\n", "\n", "。", " ", ""],
    )
    texts = splitter.split_text(text)
    return [Chunk(content=t, metadata={"chunk_method": "fallback"}) for t in texts]

LLM 分块的价值不只是"切得更准"：

  • 摘要和主题标签是天然的检索增强信号。可以把 summary 也做 Embedding 参与检索，把 topics 作为元数据过滤条件
  • 对结构模糊的文档效果显著。扫描件 OCR 出来的纯文本、没有标题层级的会议记录、口语化的培训材料——这些文档用规则分块效果差，但 LLM 能理解语义
  • 研究发现，arxiv 上的一篇实验（Can LLMs Replace Humans During Code Chunking?）表明，LLM 选择的分割点与人类专家高度一致，且 LLM 分块后下游任务的事实性提高了约 20%

成本控制： 以 GPT-4o 为例，处理一篇 5000 字的中文文档约消耗 3000-4000 Token，成本约 ¥0.15-0.2。百篇文档一次性入库约 ¥15-20。用 GPT-4o-mini 可以降到十分之一。对于文档量有限的场景，这个成本完全可接受。

LLM 分块的局限：

问题说明
长文档需分段处理超出上下文窗口的文档需要先粗切再 LLM 细分，引入了新的分块决策
输出格式不稳定即使在 Prompt 中严格要求 JSON，LLM 偶尔仍会输出非标准格式，需要降级兜底
大规模场景不经济上万篇文档的全量处理成本和耗时都不可忽视
可复现性差同一文档多次 LLM 分块可能得到不同的结果

4.2 分块策略选型#

策略语义完整性成本适用场景
固定长度差零快速验证、非结构化文本
递归字符中零通用场景的默认选择
结构感知好零有标题层级的结构化文档（推荐）
语义分块好中高精度需求、文档量小
LLM 辅助最好中-高文档量有限、结构模糊、对质量要求高

实践建议： 优先使用结构感知分块。对于纯平文本（无标题层级），降级到递归字符分块。如果文档量有限且对检索质量要求高，LLM 辅助分块是最优选择——它的摘要和主题标签产出是其他方案做不到的附加收益。语义分块作为离线评估的基准线，用来衡量其他方案的效果差距。

4.3 元数据保留策略#

每个 chunk 除了文本内容，还应该携带元数据用于后续检索过滤：

@dataclass
class Chunk:
    content: str
    metadata: dict = field(default_factory=dict)

# 推荐的元数据字段
{
    "source_file": "docs/api-design.pdf",     # 来源文件
    "page_number": 12,                         # 页码
    "heading_path": "第三章 > 3.2 认证方案",    # 标题路径
    "doc_type": "pdf",                         # 文档类型
    "department": "技术部",                     # 业务标签
    "created_at": "2026-01-15",                # 文档日期
    "chunk_index": 5,                          # 块在原文中的序号
}

这些元数据在检索阶段的作用：

  • 过滤：用户只查"技术部"的文档时，直接在元数据上过滤，不需要做语义匹配
  • 溯源：回答问题时可以展示来源文档和页码
  • Reranking：标题路径与 query 的匹配度可以作为排序信号

五、向量化与入库#

分块完成后，需要把每个 chunk 转成向量，存入向量数据库。

5.1 Embedding 模型选型#

模型维度中文效果特点
BAAI/bge-large-zh-v1.51024优秀中文场景首选，MTEB 排名高
BAAI/bge-m31024优秀多语言 + 多粒度，支持稠密+稀疏+ColBERT
text-embedding-3-large（OpenAI）3072良好无需本地部署，按 Token 计费
bce-embedding-base_v1768良好网易出品，中文优化

自部署推荐 bge-m3，它同时支持稠密向量和稀疏向量（BM25 语义），可以一步到位实现 Hybrid RAG。

5.2 批量向量化#

from sentence_transformers import SentenceTransformer
from typing import Iterator

model = SentenceTransformer("BAAI/bge-m3")

def embed_chunks(chunks: list[Chunk], batch_size: int = 64) -> list[tuple[Chunk, list[float]]]:
    """批量向量化，控制显存使用"""
    results = []
    for i in range(0, len(chunks), batch_size):
        batch = chunks[i : i + batch_size]
        texts = [c.content for c in batch]
        # bge 模型推荐在文本前加指令前缀
        texts = ["Represent this sentence for searching relevant passages: " + t for t in texts]
        embeddings = model.encode(texts, normalize_embeddings=True, show_progress_bar=False)
        for chunk, emb in zip(batch, embeddings):
            results.append((chunk, emb.tolist()))
    return results

注意事项：

  • BGE 系列模型在查询时需要在 query 前加 "Represent this sentence..." 前缀，但文档入库时的文本不需要加——只在检索时的 query 加
  • normalize_embeddings=True 让向量归一化，后续用余弦相似度时直接点积即可
  • batch_size 根据显存调整，GPU 24G 一般可以到 128

5.3 Milvus Collection 设计#

from pymilvus import MilvusClient, DataType

client = MilvusClient(uri="http://localhost:19530")

COLLECTION_NAME = "knowledge_base"

def create_collection():
    """创建 Collection，定义字段和索引"""
    schema = client.create_schema(auto_id=True)

    # 主键
    schema.add_field("id", DataType.INT64, is_primary=True)
    # 向量字段（维度与 Embedding 模型对齐）
    schema.add_field("vector", DataType.FLOAT_VECTOR, dim=1024)
    # 文本内容
    schema.add_field("content", DataType.VARCHAR, max_length=8192)
    # 元数据字段
    schema.add_field("source_file", DataType.VARCHAR, max_length=512)
    schema.add_field("page_number", DataType.INT64)
    schema.add_field("heading_path", DataType.VARCHAR, max_length=512)
    schema.add_field("doc_type", DataType.VARCHAR, max_length=32)
    schema.add_field("department", DataType.VARCHAR, max_length=64)
    # sparse 向量（用于 Hybrid 检索的 BM25 语义向量）
    schema.add_field("sparse_vector", DataType.SPARSE_FLOAT_VECTOR)

    # 索引配置
    index_params = client.prepare_index_params()
    index_params.add_index(
        field_name="vector",
        index_type="IVF_FLAT",        # 生产环境建议 HNSW
        metric_type="COSINE",
        params={"nlist": 1024},
    )
    index_params.add_index(
        field_name="sparse_vector",
        index_type="SPARSE_INVERTED_INDEX",
        metric_type="IP",
    )

    client.create_collection(
        collection_name=COLLECTION_NAME,
        schema=schema,
        index_params=index_params,
    )

设计考量：

  • IVF_FLAT vs HNSW：IVF_FLAT 内存占用更小，适合千万级以下数据量；HNSW 查询更快但内存翻倍。生产环境建议先 IVF_FLAT，查询延迟不满足时再切 HNSW
  • sparse_vector 字段：如果使用 bge-m3，它同时输出稠密和稀疏向量。稀疏向量支持关键词级别的精确匹配，与稠密向量的语义匹配互补
  • 元数据字段：用独立列而非 JSON，方便做标量过滤（如 department == "技术部"）

5.4 批量写入#

def insert_chunks(embedded_chunks: list[tuple[Chunk, list[float]]]):
    """批量写入向量数据库"""
    data = []
    for chunk, vector in embedded_chunks:
        row = {
            "vector": vector,
            "content": chunk.content,
            "source_file": chunk.metadata.get("source_file", ""),
            "page_number": chunk.metadata.get("page_number", 0),
            "heading_path": chunk.metadata.get("heading_path", ""),
            "doc_type": chunk.metadata.get("doc_type", ""),
            "department": chunk.metadata.get("department", ""),
        }
        # 如果有 sparse 向量（bge-m3 场景）
        if "sparse_vector" in chunk.metadata:
            row["sparse_vector"] = chunk.metadata["sparse_vector"]
        data.append(row)

    # 分批写入，避免单次请求过大
    BATCH_SIZE = 500
    for i in range(0, len(data), BATCH_SIZE):
        client.insert(collection_name=COLLECTION_NAME, data=data[i : i + BATCH_SIZE])

    # 写入后刷新，确保数据可检索
    client.flush(collection_name=COLLECTION_NAME)

六、生产级 Pipeline 设计#

把上述环节组装成端到端的文档处理 Pipeline 时，需要考虑几个生产环境的关键问题。

6.1 Pipeline 组装#

import hashlib
from pathlib import Path
from typing import Callable

# 格式 → 解析器的注册表
PARSERS: dict[str, Callable] = {
    ".pdf": parse_pdf,
    ".docx": parse_docx,
    ".xlsx": parse_excel,
    ".pptx": parse_pptx,
    ".html": parse_html,
    ".md": parse_html,  # Markdown 转 HTML 后用同一套逻辑
}

class DocumentPipeline:
    def __init__(self, chunk_size: int = 800, embed_batch_size: int = 64):
        self.cleaner = TextCleaner()
        self.chunk_size = chunk_size
        self.embed_batch_size = embed_batch_size

    def process_file(self, file_path: str) -> int:
        """处理单个文件，返回写入的 chunk 数"""
        ext = Path(file_path).suffix.lower()
        parser = PARSERS.get(ext)
        if not parser:
            raise ValueError(f"不支持的文件格式: {ext}")

        # 1. 解析
        doc = parser(file_path)

        # 2. 清洗
        for section in doc.sections:
            section.content = self.cleaner.clean(section.content)

        # 3. 分块
        chunks = structure_aware_chunk(doc.sections, max_chunk_size=self.chunk_size)

        # 补充元数据
        file_hash = _file_hash(file_path)
        for i, chunk in enumerate(chunks):
            chunk.metadata.update({
                "source_file": file_path,
                "doc_type": ext.lstrip("."),
                "file_hash": file_hash,
                "chunk_index": i,
            })

        # 4. 向量化
        embedded = embed_chunks(chunks, batch_size=self.embed_batch_size)

        # 5. 入库
        insert_chunks(embedded)

        return len(chunks)


def _file_hash(file_path: str) -> str:
    """计算文件 MD5，用于增量更新"""
    h = hashlib.md5()
    with open(file_path, "rb") as f:
        for block in iter(lambda: f.read(8192), b""):
            h.update(block)
    return h.hexdigest()

6.2 增量更新#

企业知识库的文档会持续更新。每次全量重建不可接受，需要增量更新机制：

class IncrementalPipeline(DocumentPipeline):
    def __init__(self, **kwargs):
        super().__init__(**kwargs)

    def process_directory(self, dir_path: str):
        """增量处理目录下的所有文件"""
        processed = 0
        skipped = 0
        updated = 0

        for file_path in Path(dir_path).rglob("*"):
            if file_path.suffix.lower() not in PARSERS:
                continue

            current_hash = _file_hash(str(file_path))

            # 查询已有的文件哈希
            existing = client.query(
                collection_name=COLLECTION_NAME,
                filter=f'source_file == "{file_path}"',
                output_fields=["file_hash"],
                limit=1,
            )

            if existing and existing[0]["file_hash"] == current_hash:
                skipped += 1
                continue

            # 文件变更或新增：先删旧数据，再重新入库
            if existing:
                self._delete_by_source(str(file_path))
                updated += 1
            else:
                processed += 1

            self.process_file(str(file_path))

        print(f"新增: {processed}, 更新: {updated}, 跳过: {skipped}")

    def _delete_by_source(self, file_path: str):
        """删除指定文件的所有 chunks"""
        client.delete(
            collection_name=COLLECTION_NAME,
            filter=f'source_file == "{file_path}"',
        )

核心思路： 用文件 MD5 作为指纹。入库时记录 file_hash，增量处理时对比——哈希不变就跳过，变了就删旧插新。

6.3 幂等性设计#

同一个文件被重复处理时，不应产生重复数据。上面的增量 Pipeline 已经保证了这一点（先删后插），但在分布式场景下，还需要加锁：

import redis

r = redis.Redis()

def process_file_idempotent(file_path: str, pipeline: DocumentPipeline):
    """带分布式锁的幂等处理"""
    lock_key = f"doc_lock:{file_path}"

    # 获取锁（超时 5 分钟，防止死锁）
    lock = r.lock(lock_key, timeout=300)
    if not lock.acquire(blocking=False):
        print(f"文件正在处理中，跳过: {file_path}")
        return

    try:
        pipeline.process_file(file_path)
    finally:
        lock.release()

6.4 异常处理与断点续传#

文档处理是批量任务，单个文件失败不应中断整个流程：

import logging

logger = logging.getLogger(__name__)

def process_directory_safe(dir_path: str, pipeline: DocumentPipeline):
    """带容错的批量处理"""
    results = {"success": 0, "failed": 0, "errors": []}

    files = list(f for f in Path(dir_path).rglob("*") if f.suffix.lower() in PARSERS)

    for i, file_path in enumerate(files):
        try:
            count = pipeline.process_file(str(file_path))
            results["success"] += 1
            logger.info(f"[{i+1}/{len(files)}] {file_path.name} → {count} chunks")
        except Exception as e:
            results["failed"] += 1
            results["errors"].append({"file": str(file_path), "error": str(e)})
            logger.error(f"[{i+1}/{len(files)}] 失败 {file_path.name}: {e}")
            # 继续处理下一个文件，不中断

    # 输出失败清单，便于手动排查
    if results["errors"]:
        logger.warning(f"失败 {results['failed']} 个文件:")
        for err in results["errors"]:
            logger.warning(f"  - {err['file']}: {err['error']}")

    return results

6.5 监控指标#

生产环境需要关注的核心指标：

指标含义告警阈值建议
解析成功率成功解析的文件数 / 总文件数< 95%
平均分块大小所有 chunk 的平均字符数偏离目标值 ±30%
向量化吞吐每分钟处理的 chunk 数根据基线设定
入库延迟从解析到写入完成的端到端时间单文件 > 60s
Chunk 总量增长知识库中的 chunk 数变化趋势异常突增需关注

七、总结#

文档处理 Pipeline 是 RAG 系统的地基，每个环节的决策都会传导到最终的检索质量：

环节核心决策影响范围
解析保留文档结构 vs 只提文本分块精度
清洗去噪力度内容质量
分块策略选择 + chunk 大小检索精度 + 上下文完整性
向量化模型选择 + 批量效率语义表达能力
入库Collection 设计 + 元数据检索灵活性
Pipeline增量更新 + 幂等性运维成本

几个实践中的经验判断：

  • 分块策略比 Embedding 模型更重要。 从 GTE 换成 BGE 带来的提升，通常不如把固定长度分块换成结构感知分块
  • 元数据是检索的倍增器。 标题路径、文档类型、部门标签这些元数据，在检索时可以大幅缩小召回范围
  • Pipeline 的健壮性比单环节的极致优化更紧迫。 先保证各类文档都能稳定入库，再逐步优化每个环节的效果
