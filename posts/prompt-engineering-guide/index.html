前言#

Prompt Engineering（提示工程）是与 LLM 交互的关键技能。同样的模型，不同的 Prompt 天差地别。这篇文章不讲玄学，只讲有理论依据、经过实践验证的方法。我们会从最简单的零样本提示出发，一步步深入到思维链、思维树等高级推理技术，并在每个环节给出可直接落地的建议。

一、零样本提示（Zero-shot Prompting）#

零样本提示就是不给任何示例，直接让模型完成任务。这是最基础也最常用的交互方式。

1.1 基本原则#

指令要清晰、具体、结构化。 模型不是读心者——越模糊的指令得到越随机的输出。

❌ 模糊: "帮我写个东西"
✅ 清晰: "写一篇 300 字的博客开头，主题是 Prompt Engineering 的重要性，目标读者是 AI 应用开发者，语气专业但不晦涩"

关键要素：

  • 角色设定：告诉模型它是什么角色（“你是一位资深 AI 工程师”）
  • 任务描述：明确要做什么（“解释 Transformer 的自注意力机制”）
  • 输出格式：指定格式（Markdown、JSON、列表等）
  • 约束条件：长度、风格、受众等限制

1.2 角色提示（Role Prompting）#

角色提示是零样本中最有效的技巧之一。设定角色相当于为模型划定了一个"思考框架"，会激活训练数据中与该角色相关的模式。

❌ 无角色: "解释一下 KV Cache"
✅ 有角色: "你是一位 AI 推理引擎工程师，向刚入行的开发者解释什么是 KV Cache，以及它为什么能加速推理"

更好的角色设定会包含身份（你是谁）+ 背景（为什么）+ 受众（对谁说）。角色越具体，输出质量越高。

1.3 输出格式控制#

LLM 天然可以输出结构化内容，但要精确控制格式需要明确说明。

请列出 Python 的 5 个 Web 框架，按以下 JSON 格式输出：
{
  "frameworks": [
    {"name": "...", "stars": number, "best_for": "..."}
  ]
}

在构建 Agent 系统时，通常会配合 JSON Schema 约束使用，但好的 Prompt 本身就能大幅降低格式错误的概率。

1.4 分隔符与结构化#

用分隔符区分 Prompt 的不同部分，能显著提升模型的上下文理解能力：

系统指令: 你是一个翻译助手，将中文翻译成英文。
用户输入: {{用户输入}}
输出格式: 只输出翻译结果，不要额外内容。

常见的分隔符包括：---、===、###、"""、XML 标签等。关键是一致性——整套 Prompt 用同一种分隔方案。

二、少样本提示（Few-shot Prompting）#

少样本提示是在 Prompt 中给出若干个示例，让模型通过类比学习任务模式。

2.1 为什么有效#

少样本提示利用了 LLM 的**上下文学习（In-Context Learning）**能力。模型在训练阶段见过大量"输入-输出"对，自然学会了从示例中推断任务模式的能力。给定几个示例，模型会：

  1. 识别示例中的共同模式
  2. 提取潜在的"规则"
  3. 将规则应用到新的输入上

2.2 示例选择策略#

示例的质量比数量重要。 3-5 个精心挑选的示例通常就足够了。

  • 代表性：示例应覆盖任务空间中的典型情况
  • 多样性：示例应涵盖不同的输入模式和边缘情况
  • 顺序：最简单的示例放在前面，复杂的在后面
  • 一致性：所有示例的格式必须严格一致
将中文商品名翻译为英文商品名。

输入: 红色加大码纯棉T恤
输出: Red XXL Cotton T-Shirt

输入: 不锈钢保温杯500ml
输出: Stainless Steel Thermos 500ml

输入: 女士真丝方巾礼盒装
输出: Women's Silk Scarf Gift Set

输入: 儿童防蓝光护目镜
输出: Children's Blue Light Blocking Glasses

2.3 示例的边缘情况#

在示例中故意包含边缘情况，能让模型更好地处理异常输入：

分类客户反馈: 正面 / 负面 / 中性

输入: 产品质量很好，物流也快
输出: 正面

输入: 和描述完全不符，差评
输出: 负面

输入: 产品还行，但包装可以改进
输出: 中性

输入: 还没收到货（示例：中性，因为没有实际体验）
输出: 中性

2.4 少样本 vs 微调#

维度少样本提示微调
成本近乎为零需要标注数据和计算资源
灵活性随时换任务一个模型只能做一个任务
效果上限依赖模型能力可超越基座模型
维护成本改 Prompt 即可需要重新训练和部署

原则：优先尝试少样本，效果不够再考虑微调。 少样本提示的成本优势是数量级的，而且对于 GPT-4 级别的模型，少样本已经能覆盖大多数场景。

三、思维链提示（Chain-of-Thought, CoT）#

思维链是 Prompt Engineering 近年最重要的突破之一。核心思想很简单：让模型在给出最终答案之前，先输出中间推理步骤。

3.1 基本原理#

标准 Prompt 要求模型直接输出答案，而 CoT Prompt 要求模型"一步一步思考"：

❌ 标准: "小明有 12 个苹果，给了小红 3 个，又买了 5 个，现在有几个？→ 14"
✅ CoT:  "小明有 12 个苹果，给了小红 3 个，又买了 5 个，现在有几个？
          一步一步分析：
          1. 小明开始有 12 个苹果
          2. 给了小红 3 个，剩下 12 - 3 = 9 个
          3. 又买了 5 个，现在有 9 + 5 = 14 个
          所以答案是 14。"

3.2 为什么 CoT 有效#

CoT 的有效性来自两个层面：

增加了推理的计算量。 Transformer 的每一层都在做模式匹配和推理。当模型输出中间步骤时，每个 Token 的生成过程都伴随着 Attention 计算，相当于为推理分配了更多的"计算预算"。直接给出答案只需要一个 Token，而 CoT 需要几十上百个 Token——这些都是推理计算，每一步都在强化推理的质量。

将隐性推理显式化。 很多任务需要的不是"知道答案"，而是"推理过程"。CoT 迫使模型把推理链展示出来，每一步都建立在前面一步的基础上，降低了"跳跃式推理"导致错误的概率。

3.3 零样本 CoT（Zero-shot CoT）#

不需要示例，只需要在 Prompt 末尾加上触发词：

用户: "一个球和一个球拍总价 1.1 元，球拍比球贵 1 元，球多少钱？"
助手: 逐一分析...

常用的触发词包括："逐一分析..."、"一步一步推理"、"Let's think step by step"、"让我们逐步思考"。

零样本 CoT 在数学推理、逻辑推理、常识推理等任务上几乎无成本地提升效果。

3.4 少样本 CoT（Few-shot CoT）#

在示例中展示推理过程，而不仅仅是输入和输出：

问: 一个会议室里有 23 个人，走了 7 个，来了 12 个，现在有多少人？
答: 原来有 23 人，走了 7 人后是 23 - 7 = 16 人，又来了 12 人后是 16 + 12 = 28 人。所以答案是 28。

问: 一本书 200 页，第一天读了 1/4，第二天读了剩下的 1/3，还有多少页没读？
答: 第一天读了 200 × 1/4 = 50 页，剩下 150 页。第二天读了 150 × 1/3 = 50 页，剩下 100 页。所以答案是 100 页。

3.5 CoT 的最佳实践#

  • 与问题匹配的推理粒度：复杂问题需要更细的步骤拆分，简单问题可以粗粒度
  • 结构化推理过程：用编号步骤或缩进组织推理
  • 验证中间步骤：在关键推理节点要求模型自我验证
  • 不要过度引导：CoT 是激发模型的推理能力，而不是限制它

3.6 CoT 的局限性#

  • 对简单任务可能过度推理：对于模型已经能直接回答的问题，CoT 不仅是浪费，还可能引入错误
  • 对模型的推理能力有依赖：小模型即使有了 CoT 也可能推理错误，因为它本身不具备相应的推理能力
  • Token 消耗显著增加：CoT 的输出 Token 数可能比直接回答多 5-10 倍，增加成本和延迟

四、思维树（Tree-of-Thought, ToT）#

思维链是单一的推理路径，思维树则是同时探索多条推理路径，并在必要时回溯。

4.1 基本原理#

ToT 将推理过程建模为树状搜索：

  1. 分解：将问题拆解为多个推理步骤
  2. 生成：在每个步骤生成多个可能的推理方向（树的分支）
  3. 评估：评估每个分支的可行性
  4. 搜索：选择最有希望的分支继续（BFS 或 DFS）
  5. 回溯：如果某个分支走不通，回到上一个决策点
问题: "设计一个高可用的支付系统"

思维链（单一路径）:
  设计支付系统 → 选择数据库 → 设计 API → ...

思维树（多条路径）:
  设计支付系统
    ├── 架构模式
    │   ├── 事件驱动
    │   ├── 微服务 + Saga
    │   └── 单体 + 异步队列
    ├── 数据存储
    │   ├── MySQL + 分库分表
    │   ├── TiDB
    │   └── PostgreSQL + Citus
    └── 容错策略
        ├── 重试 + 幂等
        ├── 死信队列 + 人工介入
        └── 最终一致性补偿

4.2 ToT 的实现方式#

在 API 层面，ToT 需要多次调用 LLM：

Step 1: "列出解决这个问题的 3 种不同方案"
  → 方案 A, 方案 B, 方案 C

Step 2: "评估方案 A 的可行性: ..."
         "评估方案 B 的可行性: ..."
         "评估方案 C 的可行性: ..."
  → 方案 A 可行但成本高, 方案 B 最佳, 方案 C 不可行

Step 3: "基于方案 B，详细展开实现步骤..."

4.3 ToT 的适用场景#

ToT 适用于需要探索和决策的复杂问题：

  • 系统架构设计（需要权衡多个备选方案）
  • 数学竞赛题（需要尝试多种解题思路）
  • 创意写作（需要构思多个情节走向）
  • 战略规划（需要考虑多种可能性及其后果）

对简单问题（事实问答、简单分类等），ToT 是过度工程。

五、进阶 Prompt 技术#

5.1 自一致性（Self-Consistency）#

多次运行相同的 Prompt，取最频繁的答案。核心假设：正确的答案会在多次采样中一致出现，而随机错误不会。

运行 3 次 CoT，取多数答案：
  Run 1: 答案 42
  Run 2: 答案 42
  Run 3: 答案 38
→ 最终答案: 42

Temperature 设置在 0.3-0.7 之间，保证采样的多样性。自一致性在数学推理和知识问答任务上效果显著。

5.2 反思与自我纠错（Self-Reflection/Recursion）#

让模型审视自己的输出并进行修正：

User: "分析这段代码的性能问题"
LLM: [初步分析]
User: "再看看有没有遗漏的内存泄漏问题"
LLM: [补充分析]

更结构化的做法是在 Prompt 中显式要求自我审查：

1. 给出你的答案
2. 审视你的答案，找出可能的问题
3. 修正这些问题，输出最终版本

5.3 生成知识提示（Generated Knowledge Prompting）#

先让模型生成相关的背景知识/事实，再基于这些知识回答问题：

Step 1: "列出关于 TCP 拥塞控制的 5 个关键概念"

Step 2: "基于以上概念，解释为什么 TCP 在大带宽高延迟网络（长肥网络）上性能不佳"

这种方法特别适合需要专业知识支撑的问答场景。

5.4 结构化 Prompt 框架#

在实践中，一个完整的 Prompt 通常包含以下部分：

[角色设定]
你是一个资深的 [角色]，擅长 [领域]。

[上下文]
以下是相关的背景信息：[...]

[任务]
需要你完成的任务是：[...]

[约束条件]
- 输出格式：[JSON / Markdown / 纯文本]
- 字数限制：[...]
- 风格要求：[专业/通俗/...]

[示例]
输入: [...]
输出: [...]

[输入]
{{用户输入}}

5.5 ReAct 模式#

ReAct（Reasoning + Acting）是 Agent 系统的核心模式，将推理和行动交替进行：

思考: 用户想知道明天的天气，我需要查天气信息
行动: 调用 get_weather(城市="北京", 日期="明天")
观察: 返回 "北京明天 20-25°C，晴"
思考: 有了天气信息，我可以生成回答
回答: 北京明天天气晴朗，气温 20-25°C，适合外出。

ReAct 是 Prompt Engineering 在 Agent 场景下的延伸——它不只是在"问-答"层面优化，而是在"思考-行动-观察"的循环中不断提高推理质量。

六、总结与最佳实践#

6.1 技术选型指南#

场景推荐技术理由
简单问答/分类零样本 + 角色够用，成本最低
有明确的模式可循少样本示例比规则更精准
数学/逻辑推理CoT显式推理过程
需要探索的方案设计ToT多路径评估
需要高可靠性自一致性投票机制降低随机误差
Agent 工具调用ReAct推理-行动循环
需要多轮修正自我反思迭代优化

6.2 调试 Prompt 的思路#

当 Prompt 效果不好时，不要随机调整。按以下顺序排查：

  1. 模型能力是否足够？ （小模型有些任务就是做不好）
  2. 指令是否足够清晰？ （是否有歧义、矛盾、缺失的信息）
  3. 示例是否合适？ （示例是否覆盖了目标场景，格式是否一致）
  4. 是否需要显式的推理过程？ （试试加"逐一分析"）
  5. 是否给出了足够的约束？ （输出格式、长度、风格等）

6.3 Prompt Engineering 的局限性#

Prompt Engineering 不是万能的。当遇到以下情况时，应考虑工程手段而非继续调 Prompt：

  • 模型持续输出格式错误 → 用 JSON Schema 约束 + 后处理校验
  • 模型缺乏特定领域的知识 → RAG 比 Few-shot 更可靠
  • 模型无法学会复杂逻辑 → 考虑微调或切换更强的模型
  • 模型输出不稳定 → 增加重试逻辑 + 自一致性

6.4 写在最后#

Prompt Engineering 正在经历一个有趣的变化：基础技术（零样本、少样本、CoT）已经成为常识，而更高级的技术（ToT、Agent 循环、工具使用）正在被框架（LangChain、Semantic Kernel、Dify）逐步封装。但底层原理不会变——理解为什么 CoT 有效、为什么角色提示能提升质量，比记住几个"魔法 Prompt 模板"更有长期价值。
