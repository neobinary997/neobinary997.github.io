Function Calling 机制解析：从原理到生产实践#

让 LLM 从"聊天机器"进化为"工具使用智能体"的关键能力。

一、什么是 Function Calling#

Function Calling（也称 Tool Use / Tool Calling）是一种让大语言模型能够调用外部函数和 API 的机制。核心要点：模型本身不执行代码——它只负责决策"该调用哪个函数、传什么参数"，真正的执行由你的应用程序完成。

整个流程可以概括为五步：

用户: "今天东京天气怎么样？"
  ↓
模型: tool_call(get_weather, location="Tokyo")   ← 模型决策，不执行
  ↓
应用: 执行 get_weather("Tokyo") → {"temp": 15, "condition": "晴"}
  ↓
模型: "今天东京天气晴朗，气温 15°C。"           ← 模型整合结果，生成自然语言

这个机制让 LLM 突破了纯文本生成的限制，能够查询实时数据、操作数据库、调用第三方 API、控制外部设备。

二、本质理解#

几个容易混淆的概念：

概念含义
Function Calling通用术语，指模型输出结构化函数调用请求
Tool Use更广义的概念，OpenAI/Anthropic 官方现用名称
Tool可被调用的能力单元，函数是最常见的 Tool 类型
Tool Call模型返回的结构化请求，包含函数名和参数 JSON
Tool Result应用执行函数后返回给模型的结果

Function Calling vs RAG： 两者都旨在为 LLM 补充外部上下文。RAG 从知识库检索静态内容；Function Calling 触发实时动作/查询。生产中最强大的模式是两者结合——RAG 提供知识基础，Function Calling 触发真实操作。

三、OpenAI 实现#

3.1 工具定义#

from openai import OpenAI

client = OpenAI()

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_weather",
            "description": "获取指定城市的实时天气信息",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {
                        "type": "string",
                        "description": "城市名称，例如 'Tokyo, Japan'"
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "温度单位"
                    }
                },
                "required": ["location"],
                "additionalProperties": False
            },
            "strict": True
        }
    }
]

定义 Schema 的要点：

  • name 清晰表达功能意图
  • description 写清楚何时调用、输入输出的含义
  • 用 enum、minimum/maximum、format 约束参数值域
  • 开启 strict: True 确保模型输出严格符合 Schema

3.2 完整调用流程#

import json
from openai import OpenAI

client = OpenAI()

# 实际函数实现
def get_weather(location: str, unit: str = "celsius") -> dict:
    """模拟天气 API 调用，生产环境替换为真实 API"""
    return {"location": location, "temperature": 15, "unit": unit, "condition": "晴"}

# 函数注册表
available_functions = {
    "get_weather": get_weather,
}

def run_conversation(user_query: str) -> str:
    messages = [{"role": "user", "content": user_query}]

    # Step 1: 发送请求，模型决定是否调用工具
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=messages,
        tools=tools,
        tool_choice="auto",  # auto | none | required
    )

    response_message = response.choices[0].message
    tool_calls = response_message.tool_calls

    # Step 2: 检查是否有工具调用
    if tool_calls:
        messages.append(response_message)  # 把 assistant 消息加入上下文

        for tool_call in tool_calls:
            func_name = tool_call.function.name
            func_args = json.loads(tool_call.function.arguments)

            # Step 3: 执行函数
            func = available_functions.get(func_name)
            result = func(**func_args) if func else {"error": f"未知函数: {func_name}"}

            # Step 4: 将结果返回给模型
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": json.dumps(result, ensure_ascii=False),
            })

        # Step 5: 再次调用模型，获取最终回复
        final_response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
        )
        return final_response.choices[0].message.content

    return response_message.content

3.3 tool_choice 参数#

# 自动模式：模型自行判断是否需要调用工具（默认）
tool_choice = "auto"

# 强制调用：模型必须调用至少一个工具
tool_choice = "required"

# 禁用工具：即使传了 tools，模型也不会调用
tool_choice = "none"

# 指定调用某个工具
tool_choice = {"type": "function", "function": {"name": "get_weather"}}

四、Anthropic Claude 实现#

Claude 的工具调用机制与 OpenAI 原理相同，但 API 格式有差异。

4.1 工具定义#

import anthropic

client = anthropic.Anthropic()

tools = [
    {
        "name": "get_weather",
        "description": "获取指定城市的实时天气信息",
        "input_schema": {
            "type": "object",
            "properties": {
                "location": {
                    "type": "string",
                    "description": "城市名称，例如 'Tokyo, Japan'"
                },
                "unit": {
                    "type": "string",
                    "enum": ["celsius", "fahrenheit"],
                }
            },
            "required": ["location"]
        }
    }
]

与 OpenAI 的关键差异：

  • 使用 input_schema 而非 parameters
  • tool_choice 为 {"type": "auto"} / {"type": "any"} / {"type": "tool", "name": "xxx"}
  • 工具结果以 tool_result content block 形式返回
  • 默认支持并行工具调用

4.2 完整调用流程#

import anthropic
import json

client = anthropic.Anthropic()

def get_weather(location: str, unit: str = "celsius") -> dict:
    return {"location": location, "temperature": 15, "unit": unit, "condition": "晴"}

def run_claude_conversation(user_query: str) -> str:
    messages = [{"role": "user", "content": user_query}]

    # Step 1: 发送请求
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )

    # Step 2: 检查是否有 tool_use
    tool_use_blocks = [
        block for block in response.content if block.type == "tool_use"
    ]

    if tool_use_blocks:
        messages.append({"role": "assistant", "content": response.content})

        tool_results = []
        for block in tool_use_blocks:
            func_name = block.name
            func_args = block.input  # Claude 直接返回 dict，无需 json.loads

            result = get_weather(**func_args)

            tool_results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": json.dumps(result, ensure_ascii=False),
            })

        messages.append({"role": "user", "content": tool_results})

        # Step 3: 获取最终回复
        final_response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            messages=messages,
        )

        text_blocks = [
            block.text for block in final_response.content if block.type == "text"
        ]
        return "\n".join(text_blocks)

    text_blocks = [block.text for block in response.content if block.type == "text"]
    return "\n".join(text_blocks)

4.3 并行工具调用#

Claude 原生支持在单次响应中返回多个 tool_use block：

# 用户: "查询东京和纽约的天气"
# Claude 会在 response.content 中返回两个 tool_use block：
#   tool_use(id="tool_01", name="get_weather", input={"location": "Tokyo"})
#   tool_use(id="tool_02", name="get_weather", input={"location": "New York"})

# 并行执行与结果返回
def execute_tools(tool_use_blocks):
    results = []
    for block in tool_use_blocks:
        result = get_weather(**block.input)
        results.append({
            "type": "tool_result",
            "tool_use_id": block.id,
            "content": json.dumps(result, ensure_ascii=False),
        })
    return results

6.2 错误处理策略#

import json
import logging

logger = logging.getLogger(__name__)

def safe_execute_tool(func_name: str, func_args: dict, registry: dict) -> str:
    """安全的工具执行，统一错误处理"""
    func = registry.get(func_name)
    if func is None:
        error_msg = f"未知函数 '{func_name}'，可用函数: {list(registry.keys())}"
        logger.warning(error_msg)
        return json.dumps({"error": error_msg})

    try:
        result = func(**func_args)
        return json.dumps(result, ensure_ascii=False, default=str)
    except TypeError as e:
        logger.error(f"参数错误 {func_name}: {e}")
        return json.dumps({"error": f"参数错误: {str(e)}"})
    except Exception as e:
        logger.exception(f"执行失败 {func_name}")
        return json.dumps({"error": f"执行失败: {str(e)}"})

6.3 工具结果过大时的处理#

工具返回大量数据会撑爆上下文窗口。处理策略：

def truncate_large_result(result: str, max_chars: int = 4000) -> str:
    """截断过大的工具结果，保留头部信息"""
    if len(result) <= max_chars:
        return result
    summary = result[:max_chars] + f"\n\n... [截断，总共 {len(result)} 字符]"
    return summary

6.4 并发工具执行#

当模型返回多个独立工具调用时，应并行执行以降低延迟：

import asyncio

async def execute_tools_parallel(tool_calls, registry):
    """并行执行多个独立的工具调用"""
    async def call_one(tool_call):
        args = json.loads(tool_call.function.arguments)
        result = await asyncio.to_thread(
            safe_execute_tool, tool_call.function.name, args, registry
        )
        return {
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": result,
        }

    return await asyncio.gather(*[call_one(tc) for tc in tool_calls])

6.5 防止幻觉：验证模型输出的参数#

def validate_tool_args(tool_call, tool_schemas: dict) -> bool:
    """验证模型返回的参数是否符合 Schema"""
    schema = tool_schemas.get(tool_call.function.name)
    if not schema:
        return False

    args = json.loads(tool_call.function.arguments)
    required = schema.get("parameters", {}).get("required", [])

    # 检查必填字段
    for field in required:
        if field not in args:
            logger.warning(f"缺少必填参数 '{field}'，tool: {tool_call.function.name}")
            return False

    return True

6.6 Token 优化#

# 优化前：每个 Tool 的完整 Schema 都占用 Token
tools = [tool_a, tool_b, tool_c, ..., tool_z]  # 20+ tools → 大量 Token 消耗

# 优化方案 1：Tool Search（OpenAI gpt-5.4+ / Anthropic）
# 模型按需检索工具定义，而非一次性加载所有

# 优化方案 2：工具分组
# 将工具分为常用和罕见两组，初次调用只传常用组
COMMON_TOOLS = [tool_a, tool_b, tool_c]
RARE_TOOLS = [tool_x, tool_y, tool_z]

# 优化方案 3：精简 Schema
# 删除不必要的 description 细节，只保留区分性描述

七、跨平台统一封装#

当应用需要同时支持多个 LLM 提供商时，可以用适配器模式统一接口：

from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import Any

@dataclass
class ToolCall:
    id: str
    name: str
    arguments: dict

class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages: list, tools: list) -> tuple[str | None, list[ToolCall]]:
        """返回 (文本回复, 工具调用列表)"""
        ...

class OpenAIProvider(LLMProvider):
    def __init__(self, api_key: str, model: str = "gpt-4o"):
        self.client = OpenAI(api_key=api_key)
        self.model = model

    def chat(self, messages, tools):
        response = self.client.chat.completions.create(
            model=self.model, messages=messages, tools=tools
        )
        msg = response.choices[0].message
        tool_calls = [
            ToolCall(id=tc.id, name=tc.function.name,
                     arguments=json.loads(tc.function.arguments))
            for tc in (msg.tool_calls or [])
        ]
        return msg.content, tool_calls

class AnthropicProvider(LLMProvider):
    def __init__(self, api_key: str, model: str = "claude-sonnet-4-6"):
        self.client = anthropic.Anthropic(api_key=api_key)
        self.model = model

    def chat(self, messages, tools):
        # 转换工具格式
        claude_tools = [
            {"name": t["function"]["name"],
             "description": t["function"]["description"],
             "input_schema": t["function"]["parameters"]}
            for t in tools
        ]
        response = self.client.messages.create(
            model=self.model, messages=messages,
            tools=claude_tools, max_tokens=4096,
        )
        text = "".join(b.text for b in response.content if b.type == "text")
        tool_calls = [
            ToolCall(id=b.id, name=b.name, arguments=b.input)
            for b in response.content if b.type == "tool_use"
        ]
        return text or None, tool_calls

八、常见问题与排查#

Q1: 模型不调用工具#

  • 检查 Tool 的 description 是否足够清晰描述使用场景
  • 检查 tool_choice 是否设为 "none"
  • 用户的问题是否确实需要工具？有时模型判断可以纯文本回答

Q2: 模型调了错误的工具#

  • 多个 Tool 的 description 太相似 → 在描述中加入"区别于 XX 函数"的说明
  • 给每个 Tool 添加使用示例（Anthropic 支持 Tool Use Examples）
  • 在 system prompt 中明确使用场景规则

Q3: 参数解析失败#

  • 开启 strict: True（OpenAI）保证 Schema 合规
  • 对 arguments 做 JSON parse 时始终用 try/except 包裹
  • 用上一节的 validate_tool_args 做二次校验

Q4: 工具调用无限循环#

九、发展趋势#

Function Calling 正在快速演化，几个值得关注的方向：

  1. Model Context Protocol (MCP)：跨提供商的标准工具协议，让同一个 Tool 定义在不同 LLM 间复用
  2. Programmatic Tool Calling（Anthropic）：模型编写 Python 脚本编排工具调用链，模型只看到最终结果，大幅节省上下文
  3. Tool Search：按需检索工具定义，解决"工具太多"的问题
  4. Streaming Tool Use：工具调用也支持流式返回，减少用户等待时间
  5. Code Execution + Tool Use 结合：模型在沙箱中同时写代码和调用工具，实现更复杂的工作流

十、总结#

Function Calling 是将 LLM 从对话机器人升级为智能体的基础设施。实践中的关键原则：

  • Schema 是契约：写得越精确，模型调用越准确
  • 错误处理是底线：模型输出不可全信，始终校验和兜底
  • 并行优于串行：独立工具调用尽量并发执行
  • Token 是成本：工具定义占用上下文，需要时再加载

掌握了这些，你就可以用 Function Calling 构建出能够查询实时数据、操作外部系统、执行复杂工作流的 AI 应用。
